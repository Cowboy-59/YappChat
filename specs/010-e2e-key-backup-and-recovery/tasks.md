## Task Details

### T001 — Backup envelope format + storage APIs + audit log + database schema

Implement FR-001, FR-003, FR-006. Create tables per project DB conventions (lowercase, no separators, UUID v7 PKs, FK names match parent + `id`): `userkeybackups` (UNIQUE on `userid`, `blob` bytea, `blobsizebytes`, `clientversion`, `createdat`, `updatedat`), `keyrecoveryattempts` (UNIQUE userid, `failedcount`, `lastfailedat`, `lockeduntil`, `windowstartedat`), `keybackupauditlog` (90-day TTL via `expiresat`, anonymised IP), `devicetrust` (UNIQUE(userid, newdeviceid), Ed25519 signature over newdevicepublickey), `keypairings` (5-minute TTL, status enum `awaiting_existing`/`completed`/`expired`). Generate migrations via `npm run db:generate`, validate with `wxkanban-agent dbpush --dry-run`. Define the versioned blob envelope (binary, base64 for transport only): [1 byte version=0x01][16 bytes salt][8 bytes Argon2 params: memory_kib u32 + iterations u16 + parallelism u16][24 bytes XChaCha20-Poly1305 nonce][N bytes ciphertext + 16-byte Poly1305 tag]. Plaintext JSON payload includes `version`, `userid` (defends against blob-swapping), `createdat`, `identitykeys[]` (one entry per registered device with `deviceid`, `privatekey`, `publickey`, `createdat`), and `groupsessionkeys[]` (`groupid`, `key`, `rotation`, `effectivefromat`). Parser MUST refuse unknown versions with a clear error. Implement endpoints: `POST /api/keybackup` (body `{ blob, clientversion }`, replaces latest, returns `{ acknowledgedat, blobsizebytes }`, 1MB hard cap → 413), `GET /api/keybackup/exists` (returns `{ exists, lastbackupat?, blobsizebytes? }` only — NEVER the blob), `DELETE /api/keybackup` (irreversible, requires explicit confirmation client-side). Implement `keybackupauditlog` writer recording every backup, recovery, pairing, and fresh-identity event with anonymised IP (last octet zeroed for v4, last 80 bits for v6), `useragent`, `deviceid` when known. Daily cleanup job purges entries past `expiresat`. `GET /api/keybackup/audit` returns the caller's last-90-days entries. Server architecture review MUST verify the server has no path to plaintext.

### T002 — Client-side crypto primitives: Argon2id KDF + XChaCha20-Poly1305 + BIP-39 recovery code generation

Implement FR-002. Wire Argon2id client-side: `argon2-browser` on web, `react-native-argon2` (via Expo plugin) on mobile. Default parameters: m_cost 64MB (`m_cost: 65536` KiB), t_cost 3, p_cost 1. Output 32 bytes. Parameters MUST be stored IN the blob (FR-001) so future defaults can change without breaking older blobs. KDF runtime target: 500ms–2s on a mid-range mobile device. Implement XChaCha20-Poly1305 via libsodium (`@noble/ciphers` on web, `react-native-libsodium` on mobile). Implement BIP-39 recovery code generation: 24 English BIP-39 words (2048-word list → 11 bits per word → 264 bits, of which 256 are key material and 8 are checksum). MUST use `crypto.getRandomValues` — `Math.random` is a hard violation that fails CI. Passphrase path: minimum 12 characters; client computes `zxcvbn` score and rejects < 3 with a strength meter explaining why. The server is NOT involved in strength validation. Both paths derive the wrapper key via Argon2id with the parameters from the blob's header (or current defaults on first backup). Export a small key-bundle (de)serialiser that wraps/unwraps the JSON plaintext payload defined in T001. Provide a clear deserialisation error when the plaintext `userid` doesn't match the authenticated caller (defence against blob-swapping at the storage layer).

### T003 — Recovery flow with server-side rate limiting + lockouts + PA notifications

Implement FR-004. `POST /api/keybackup/begin-recovery` — authenticated, returns the user's `userkeybackups.blob`. Concurrent recovery requests MUST be serialised at the database level via row-locking on `keyrecoveryattempts.userid` — otherwise an attacker firing 100 parallel requests could outrun the counter. Returns HTTP 429 `{ error: 'recovery_locked', retryafterseconds }` when `failedcount >= 5` inside the 24-hour sliding window, or while `lockeduntil > now()`. `POST /api/keybackup/recovery-succeeded` (authenticated, no key material in body) resets `failedcount` to 0 and writes audit `recovery_succeeded`. `POST /api/keybackup/recovery-failed` (authenticated) increments `failedcount` and writes audit `recovery_failed` — purely a telemetry signal, NOT the rate-limit gate. **Lockout policy**: 5 failed in 24h → set `lockeduntil = now() + 24h`. 10 failed in 7d → set `lockeduntil = now() + 7d`. Lockout periods are absolute server-side decisions; the client cannot override. **PA alerts** via spec 002 FR-017 `postPANotification` with `bypassQuietHours: true`, `callerscope: 'keybackup-recovery'`, type `keybackup_recovery_alert`: after 3 failed in 1 hour → previewtext `"3 incorrect recovery attempts on your account. If this wasn't you, change your auth credentials."`; after lockout triggers → stronger previewtext `"Recovery locked due to repeated failures. Possible attack — review your account."` PA notification reaches user within 60s of the third failed attempt. Lockout state surfaces via `RecoveryLockedScreen` showing remaining duration and the fresh-identity escape hatch. Every request path (including 429s and expired-pairing) writes an audit row so abuse patterns are observable.

### T004 — Cross-device handoff: QR-code pairing protocol with X25519 + HKDF + XChaCha20-Poly1305

Implement FR-005. Pairing protocol: (1) new device generates an ephemeral X25519 keypair `(eph_pub_new, eph_priv_new)`; (2) `POST /api/keybackup/pairing/start` (returns 401 if unauthenticated per spec 011 FR-016) creates a `keypairings` row and returns `{ pairingid, nonce, wsscopepath: 'pairing:{pairingid}', expiresin: 300 }`. The pairing nonce is signed with the server's auth secret so the existing device verifies the QR was issued by this server (prevents pairing-QR spoofing across deployments). (3) new device displays a QR encoding `{ pairingid, nonce, eph_pub_new, wsscopepath }` PLUS an HOTP-derived 8-digit fallback code (computed from the nonce so the existing device can verify without a server round-trip). (4) existing device scans QR (or types fallback code). UI on existing device shows `"Pair {target user}'s device"` so user verifies the target before transferring. (5) existing device generates its own ephemeral keypair `(eph_pub_old, eph_priv_old)`, computes `K = X25519(eph_priv_old, eph_pub_new)`, derives the wrap key via HKDF-SHA256. (6) existing device encrypts the full key bundle JSON (same shape as FR-001 plaintext) with XChaCha20-Poly1305 + fresh nonce, then `POST /api/keybackup/pairing/:id/deliver` body `{ eph_pub_old, ciphertext, nonce }`. Server forwards the payload over WS scope `pairing:{pairingid}` and stores NOTHING — fire-and-forget. If the new device isn't subscribed, return HTTP 404 `{ error: 'new_device_not_listening' }`. WS subscription auth (spec 003 FR-002) MUST check the pairing nonce AND `keypairings.userid = currentuserid` AND a valid session — the new device must complete sign-in BEFORE its subscription is accepted. (7) new device receives over WS, computes `K = X25519(eph_priv_new, eph_pub_old)`, derives the same HKDF key, decrypts, writes keys to `SecureKeyStore`, generates its own NEW identity keypair (the transferred bundle gives history-decrypt access; the new device gets its own new public key for going-forward sends), publishes the new public key to `userencryptionkeys` with `trustedby: existingdeviceid`. (8) existing device signs a trust certificate (Ed25519 over `newdevicepublickey`) and posts to `POST /api/keybackup/devicetrust`. `GET /api/keybackup/devicetrust` lists trust certs for the caller's devices. Pairing fails gracefully on either-side dropout — new device shows `"Pairing timed out"` and offers restart. No partial state left on the server. Pairing single-use, 5-minute TTL; per-minute cleanup job purges expired rows. End-to-end pairing ≤ 30s. Server compromise CANNOT extract the transferred bundle from this flow. **Cross-spec deviceid contract**: the value in `keypairings.id` matches the new device's `userencryptionkeys.deviceid` (spec 001) ↔ `mobiledevices.deviceid` (spec 008) ↔ `pushtokens.deviceid` (spec 009) ↔ `devicesessions.deviceid` (spec 011 FR-014).

### T005 — Group session key inclusion + debounced re-upload + 1MB-cap pruning

Implement FR-007. The backup blob's plaintext `groupsessionkeys[]` array contains one entry per group the user is currently a member of: `{ groupid, key, rotation, effectivefromat }`. Multiple historical session keys per group are retained — when a key rotates, the OLD key is kept in the bundle UNTIL messages encrypted with it have all been purged per the user's retention policy (spec 001 FR-012). On group membership change OR group session-key rotation, the client immediately schedules a re-encrypt-and-re-upload. **Debounce**: at most one re-upload per 60s per user (prevents thrashing if many group operations happen at once). The 60s debounce window is the only allowed staleness — outside that window the server-stored blob is ≤ 60s out of date with current key state. On recovery, the client extracts every group session key into the local key store and historical group messages decrypt successfully with the correct generation key. **1MB cap pruning** (FR-007 edge case): if the recomputed blob would exceed the 1MB cap (in practice > ~1000 group session keys), the client prunes oldest-rotation keys from groups where the user no longer holds membership AND emits a warning via `KeyBackupOutOfDate` banner. If even after pruning the blob is over the cap, surface a structured error to the UI and block further re-uploads until the user prunes manually from settings. Keys for groups where the user left AND whose messages have been purged are NEVER carried forward.

### T006 — Passphrase rotation + fresh-identity escape hatch + setup nudge banners

Implement FR-008, FR-009, FR-010. **Rotation** (FR-008): `KeyBackupSettings` UI offers **Change passphrase** and **Generate new recovery code**. Client requires the CURRENT secret first, validated by attempting to decrypt the existing blob LOCALLY (server is NOT involved in this validation). After local validation, the client derives a new wrapper key from the new secret, re-encrypts the existing key-bundle plaintext, and posts via `POST /api/keybackup` — old blob is overwritten, no version history. Audit row `backup_replaced`. Identity keys are unchanged; other YappChat clients see no change. **Fresh identity** (FR-009): after 5 failed recovery attempts OR proactively at any time during recovery, the UI offers **Start a fresh identity** via `FreshIdentityConfirm`. Dialog text MUST be explicit: `"Your old messages will remain encrypted. They will never be readable. Your contacts will see this as a new identity."` On confirmation, client generates a new identity keypair, registers it in `userencryptionkeys` WITHOUT a trust certificate, and deletes the user's backup blob via `DELETE /api/keybackup`. Irreversible. Audit row `fresh_identity_started`. Other YappChat clients on next message-send to this user check `userencryptionkeys`, see an untrusted new key, and surface a warning: `"{name}'s security key changed. This may mean they re-installed YappChat, but it could also mean someone is impersonating them. Verify out-of-band before sending sensitive info."` Old encrypted messages remain on the server, take up storage, and will eventually be purged per the user's retention policy. **Setup nudges** (FR-010): if `userkeybackups` has no row for the user, `KeyBackupReminder` banner appears once every 7 days; dismissal sets `lastreminderdismissedat` and the next reminder fires exactly 7 days later. Separately, `KeyBackupOutOfDate` banner appears when the user HAS set up backup AND `updatedat` is > 14 days old AND the user has joined ≥ 1 new group OR rotated keys since the last backup — one click re-uploads the current bundle. Suppress reminders during onboarding and during active conversations. Users can disable reminders entirely in settings → security, replacing the banner with a passive "Set up recovery" link.

### T007 — Frontend onboarding + setup surface: KeyBackupOnboarding, RecoveryCodeReveal, RecoveryCodeConfirmation, PassphraseSetup, banners

Build the onboarding + setup React components under `packages/ui/src/keybackup/`. `KeyBackupOnboarding.tsx` — multi-step modal triggered by the shell on first launch after key generation. Steps: explainer → choose **Generate recovery code** (recommended) or **Set a passphrase** → confirm → store. Onboarding is skippable but FR-010's reminder runs every 7 days until completed. `RecoveryCodeReveal.tsx` — displays the 24 words once on a screenshot-blocked surface: `FLAG_SECURE` on Android, `UIScreen.main.isCaptured` screen-recording detection on iOS. Web cannot block screenshots; instead show a prominent warning `"Don't screenshot — write down or print"`. Offers Print and Download-PDF buttons. PDF includes the 24 words in a readable grid, date generated, brief instruction, and a friendly reminder (`"Store this where you store other important documents. YappChat cannot recover this for you."`). `RecoveryCodeConfirmation.tsx` — grid prompting the user to type 4 randomly chosen words from their 24 to confirm memorisation BEFORE the backup is finalised. Two failures unlock the print/PDF download option. `PassphraseSetup.tsx` — passphrase input with `zxcvbn`-driven strength meter; rejects scores < 3 with an explanation; re-type for confirmation; shows the explicit warning `"If you forget this, your history cannot be recovered. We recommend the recovery code instead."` `KeyBackupReminder.tsx` — top-of-app banner: `"Set up recovery so you don't lose your messages."` Dismissible; reappears every 7 days per FR-010. `KeyBackupOutOfDate.tsx` — one-click banner: when the stored blob is stale (> 14 days AND group/key changes since). All components MUST integrate with the existing `SecureKeyStore` (spec 008 FR-004) — storing keys in `localStorage`, plain IndexedDB without wrapping, or `AsyncStorage` is a hard violation.

### T008 — Frontend recovery + pairing + settings surface: KeyRecoveryFlow, pairing UIs, settings, audit view

Build the recovery, pairing, and settings React components under `packages/ui/src/keybackup/`. **Recovery**: `KeyRecoveryFlow.tsx` — driven by the shell on launch when no key is present AND no other trusted device exists. Recovery method picker (recovery code vs passphrase), input grid, decryption attempt (calls `POST /api/keybackup/begin-recovery`, decrypts locally with Argon2id-derived key, on success calls `POST /api/keybackup/recovery-succeeded`, on failure calls `POST /api/keybackup/recovery-failed`), error states (wrong code, locked, no backup). `RecoveryLockedScreen.tsx` — shown when server returns 429: explains lock duration (parse `retryafterseconds`) and surfaces the fresh-identity alternative. `FreshIdentityConfirm.tsx` — final-step modal before destroying access; explicit, irreversible language per FR-009. **Pairing**: `PairNewDeviceQR.tsx` — shown on the NEW device: calls `/pairing/start`, generates ephemeral X25519 keypair, renders QR encoding `{ pairingid, nonce, eph_pub_new, wsscopepath }` + the 8-digit HOTP fallback code, subscribes to the WS scope `pairing:{pairingid}`, waits for delivery. `PairExistingDeviceScanner.tsx` — shown on the EXISTING device: camera scanner on mobile OR 8-digit code input on web; verifies the pairing nonce server-signature, shows `"Pair {target user}'s device"`, then runs the HKDF-derive + XChaCha20-Poly1305-encrypt flow and posts to `/pairing/:id/deliver`. `PairingProgress.tsx` — both devices show live status: `"Waiting for partner device"` → `"Encrypting"` → `"Transferring"` → `"Verifying"` → `"Done"`. Tolerates mobile background suspension on the existing device side — new device shows `"Waiting for sender to come back"` and resumes when foregrounded. 5-minute hard window — on expiry both ends restart from a fresh QR. **Settings**: `KeyBackupSettings.tsx` — status of current backup, **Change passphrase** / **Generate new recovery code** actions wired to FR-008 rotation, **View audit log**, **Delete backup** (irreversible confirmation), **Disable reminders**. `DeviceTrustList.tsx` — lists trusted devices with last-seen, signing relationship, and a revoke action. `KeyBackupAuditView.tsx` — renders entries from `GET /api/keybackup/audit` with anonymised IPs and friendly descriptions.
