## Task Details

### T001 — AIChatPanel root layout: slide-in animation + entry-point subscription + state preservation across open/close

Implement FR-001. Build `AIChatPanel.tsx` at `src/ui/components/chat/AIChatPanel.tsx` — fixed-position overlay anchored to the right edge of the viewport. **Desktop (≥ 768px)**: `position: fixed; right: 0; top: 0; height: 100vh; width: 33vw`. **Mobile (< 768px)**: `width: 100vw`, covers full screen. Panel opens with a slide-in animation from the right (200ms ease-out) and closes with a slide-out to the right. **Entry-point subscription**: subscribe to the `OpenAIChatPanel` action dispatched by spec 002 FR-001 — there is NO direct coupling to spec 002 components beyond this typed action. Two sources: `{ source: 'bubble', notificationId }` opens panel scrolled to and highlighting the linked message; `{ source: 'avatar' }` opens panel at the most recent conversation position. The panel is triggered ONLY by these two — there is NO standalone `/chat` route. **Desktop**: clicking outside the panel (on the underlying YappChat view) closes it; the underlying view remains interactive and receives clicks normally. **Mobile**: panel includes a Back button (top-left) that closes it and returns to the underlying view. **State preservation**: scroll position, active session, partial draft input, and attachment preview state are all retained when the panel is closed and reopened — implement via a `ChatPanelState` context held above the open/close mount. Build `AIChatLayout.tsx` as the inner split: `ChatSessionStrip` on the left edge of the panel (80px collapsed showing session avatar/icon stubs + a + button, 220px expanded showing the full `ChatSessionSidebar`, toggle via chevron button), `ChatMain` taking remaining width. `ChatMain.tsx` stacks `ChatMessageThread` above `ChatInputArea`. On first open with no sessions: `ChatEmptyState.tsx` shows PA avatar + name + prompt `"What can I help you with?"` centred in the conversation area. The PA avatar comes from spec 007 `<AvatarDisplay size={128}>` reading `GET /api/avatar/current`.

### T002 — Session management: sidebar with new/rename/delete/search/switch wired to spec 002 APIs

Implement FR-002. Build `ChatSessionSidebar.tsx` (search input at top + `+ New` button + scrollable `ChatSessionList`), `ChatSessionList.tsx` (filtered/sorted list of `ChatSessionItem`), `ChatSessionItem.tsx` (single row showing session `name`, last message preview truncated to 60 chars, relative timestamp `"3 hours ago"`, active highlight), `ChatSessionSearch.tsx` (live client-side filter over already-loaded sessions list, highlights matched text in both session names AND last-message-preview content). Wire to spec 002 endpoints (read-only — spec 005 calls them, does NOT define them): `GET /api/pa/sessions` populates the session list on panel mount (sorted by `lastmessageat DESC`). **+ New**: button at top of sidebar creates a session via `POST /api/pa/sessions`; new session name defaults to current date/time formatted as `"May 10, 4:32pm"` (use user locale); session becomes active immediately and the thread clears. **Rename**: double-click any session name in the sidebar makes it editable inline; Enter or blur saves via `PATCH /api/pa/sessions/:id`; Escape cancels (no save). **Delete**: right-click on desktop or swipe-left on mobile shows a Delete action; confirmation dialog required (`"Delete this session? This cannot be undone."`); on confirm `DELETE /api/pa/sessions/:id` then remove from local list (the spec 002 endpoint soft-deletes; hard-delete happens 30 days later via spec 002's retention job). **Switch**: clicking a session in the sidebar makes it active and loads its full message history from `GET /api/pa/sessions/:id/messages` — cursor-based pagination matching spec 002 FR-001 — within 1 second. While loading, show a skeleton placeholder in the thread (NOT a spinner that blocks the panel). Session list shows session name, last message preview, relative timestamp.

### T003 — Streaming SSE message rendering: token-by-token bubble growth + StreamingCursor + auto-scroll + interruption recovery

Implement FR-003. `POST /api/pa/sessions/:id/messages` is called with `Accept: text/event-stream`. The response is an SSE stream of token deltas. **Build `ChatMessageThread.tsx`** — virtualised scrollable message list using `@tanstack/react-virtual` (or `react-virtual`) — required because sessions with 200+ messages cause scroll performance issues in non-virtualised implementations. Renders `ChatMessage` per row. Auto-scrolls to bottom on new messages UNLESS the user has manually scrolled up (detect via a `scrollAnchorRef` — if the user is within 100px of the bottom when the new message arrives, auto-scroll; otherwise leave scroll position alone and show a `"Jump to latest"` button). `ChatMessage.tsx` — user messages right-aligned (no avatar); PA messages left-aligned with PA avatar (`<AvatarDisplay size={24} state={state}>` from spec 007). `ChatMessageRenderer.tsx` — renders a PA message's content as plain streaming text DURING the stream, then re-renders with full markdown (T004) AFTER stream ends — the streaming phase uses plain text for performance; the post-stream final render applies the full markdown/code/card pipeline. **Each SSE delta is appended to the streaming bubble in real time** — the bubble grows as tokens arrive. While streaming, render `StreamingCursor.tsx` (blinking caret) at the end of the last token; it disappears when the stream ends. First streaming token MUST appear within 1 second of sending. Streaming is visually smooth — NO layout thrash or scroll jump as the bubble grows. **Interruption recovery**: if the stream is interrupted (network error / disconnect / parse failure), the partial response is preserved on screen with a `"Response interrupted — retry?"` action button that re-POSTs the original user turn with the same payload. The `Stop generating` button (T005) calls `AbortController.abort()` on the SSE stream — it does NOT call a server-side cancel endpoint; the partial response is persisted as-is by spec 002. The thread MUST handle `tool_call_*` and `subagent_spawned` deltas inline (the cards themselves are built in T004).

### T004 — Rich content rendering: markdown + code blocks + ToolCallCard + SubagentCard + SkillResultCard + DashboardCard + ScheduleCard + FileAttachmentChip + SuggestedReplyChip

Implement FR-004. Wire `react-markdown` + `remark-gfm` plugin for GitHub-Flavored Markdown (tables, strikethrough, task lists). Headings, bold, italic, bullet/numbered lists, blockquotes, horizontal rules render via `react-markdown`. **Code blocks** — syntax-highlighted via `highlight.js` through `rehype-highlight` plugin (lighter than Prism). Language label shown in the block header; copy-to-clipboard button in the top-right of every code block (uses `navigator.clipboard.writeText`). **Inline code** — rendered in a monospace font with a subtle background colour. **Tables** — rendered as HTML tables with a horizontal-scroll wrapper for wide tables (`overflow-x: auto`). **`ToolCallCard.tsx`** — inline card shown when the PA invokes a skill. Shows skill label + spinner while running; collapses to a one-line summary on completion (e.g., `"Fetched 47 messages from Slack #engineering"`). Expandable to show raw input/output for debugging. Wires to spec 002 `tool_call_*` SSE deltas. **`SubagentCard.tsx`** — inline card when the PA spawns a subagent. Shows subagent avatar (spec 007 `<AvatarDisplay size={24}>`), name, current status badge (`working` / `waiting_for_input` / `completed`), and a link to the agent's channel. Wires to spec 002 `subagent_spawned` and `subagent.status` updates. **`SkillResultCard.tsx`** — renders a structured skill result as a formatted card rather than raw JSON when the skill response includes a `resulttype` field. Card type determined by a registry at `src/lib/skill-result-renderers.ts` — third-party skills can register custom renderers. Default mappings: `calendar_event` → title, date/time, location, attendees, Add-to-Calendar button; `jira_issue` → issue key, title, status badge, assignee, priority, link; `file` → filename, size, Download button (calls `GET /api/chat/attachments/:id` via T007); `weather` → city, temperature, condition icon, high/low; `generic_table` → sortable HTML table. **DashboardCard and ScheduleCard** from spec 002 are rendered inline when the PA returns those payloads — import and pass through. **`FileAttachmentChip.tsx`** — files attached to user messages shown as a chip with file icon, name, formatted size. Click downloads/previews the file via `GET /api/chat/attachments/:id` (T007). **`SuggestedReplyChip.tsx`** — up to 3 one-click suggested follow-ups appear below each PA response, generated by the PA alongside its reply. Clicking one fills the input AND immediately sends that message.

### T005 — Message input area: ChatTextInput + VoiceInputButton + AttachmentButton + SendButton + Stop-generating + character limit

Implement FR-005. Build `ChatInputArea.tsx` at the bottom of the panel — orchestrates `ChatTextInput`, `VoiceInputButton`, `AttachmentButton`, `SendButton`, `SuggestedReplyChips` (from T004), `AttachmentPreviewBar`, and `CharacterCounter`. **`ChatTextInput.tsx`** — multi-line auto-expanding textarea (auto-expands up to 6 lines then scrolls). Reads `enterkeysends` from `userchatpreferences` (T007): when `true` (default), Enter sends and Shift+Enter inserts a new line; when `false`, Enter inserts a new line and Cmd/Ctrl+Enter sends. **Cmd+Enter (Mac) / Ctrl+Enter (Windows) ALWAYS sends** regardless of Enter key mode. **Character limit**: 4,000 characters per message; counter shown when within 500 chars of limit. **`SendButton.tsx`** — send-arrow icon active ONLY when input is non-empty or a file is attached. Disabled state when input is empty AND no attachment. While the PA is streaming a response (T003), the SendButton is replaced by a **Stop generating** button — clicking it calls `AbortController.abort()` on the SSE stream. **`VoiceInputButton.tsx`** — microphone icon. Triggers the browser `SpeechRecognition` API (Chrome/Edge only — hidden on unsupported browsers with a tooltip explaining why). Pulses while recording. Transcription populates `ChatTextInput` in real time as the user speaks. Reads `voiceautosend` from preferences (T007): when `true`, auto-send 1.5s after silence; when `false` (default), user must press Send manually. Voice activation must complete within 100ms of mic-button click (spec 007 T003 listening animation depends on this fast signal). **`AttachmentButton.tsx`** — paperclip icon. Opens file picker accepting PDF, DOCX, XLSX, PNG, JPG, TXT only. Max 10MB per file, max 5 files per message — enforced BOTH client-side (pre-upload) AND server-side (T007). Uploads via `POST /api/chat/attachments` (T007); on success returns an `attachmentid` included in the next message payload. **`AttachmentPreviewBar.tsx`** — shown above the input when files are attached; list of `FileAttachmentChip` items with individual remove buttons. Files in the preview bar are uploaded but not yet attached to a sent message; removing a chip cancels that upload reference.

### T006 — Keyboard shortcuts + command palette + KeyboardShortcutsOverlay

Implement FR-006. The AI Chat MUST be fully keyboard-navigable with discoverable shortcuts. Implement these global shortcuts when the panel is open (use a single keyboard-handler at the panel level — NOT per-component): **Cmd/Ctrl + K** → create a new session (calls the same handler as the `+ New` button in T002). **Cmd/Ctrl + Shift + K** → open session search / switcher (focuses `ChatSessionSearch` input from T002). **Cmd/Ctrl + Enter** → send message (always, regardless of Enter-key mode). **Escape** → cancel editing a session name (T002) / dismiss any open dialogs / collapse Studio back to chat when Studio is active (T008) — Escape does NOT close the entire panel. **`/` at the start of input** → opens `ChatCommandPalette` — quick-action palette listing `/new` (new session), `/clear` (clear current session draft), `/export` (open session export — calls T007 export flow), `/settings` (open chat preferences modal — preferences from T007). Selecting a palette item executes the action. **`?` when input is empty and focused** → shows `KeyboardShortcutsOverlay.tsx` — modal listing all shortcuts grouped by category (Session, Composition, Navigation). Closes on Escape or click-outside. All shortcuts are shown in tooltips on hover of the relevant buttons (e.g., the `+ New` button's tooltip shows `"New session (Cmd+K)"`). Implement OS-aware tooltip rendering — Mac shows `⌘+K`, Windows/Linux shows `Ctrl+K`. Keyboard shortcuts MUST NOT fire while focus is inside a text input that has not yet started with `/` — except for the universal send shortcut (Cmd/Ctrl+Enter) which works from inside the textarea.

### T007 — Backend data model + APIs: chatattachments + userchatpreferences + attachment upload/download + preferences endpoints

Implement the spec 005-owned data model and API surface. Create tables per project DB conventions (lowercase, no separators, UUID v7 PKs, FK names match parent + `id`): **`chatattachments`** (`userid`, `sessionid` FK → spec 002 `assistantsessions.id`, `messageid` nullable FK → spec 002 `assistantmessages.id` set when the message is sent — spec 002 `assistantmessages.attachmentids` carries the reverse reference as a uuid array, `filename` original, `mimetype` e.g. `application/pdf`, `sizebytes`, `storagekey` internal path NEVER exposed as a public URL, `createdat`, `expiresat` set to session retention expiry — purged when the parent session is hard-deleted 30 days after `assistantsessions.deletedat`). **`userchatpreferences`** (UNIQUE userid, `enterkeysends` boolean default true, `voiceautosend` boolean default false, `layout` text `spacious`|`compact` for message density, `updatedat`). Generate migrations via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. **Endpoints**: `POST /api/chat/attachments` — multipart upload. Validates: accepted formats PDF/DOCX/XLSX/PNG/JPG/TXT only, ≤ 10MB per file, max 5 files in flight per session for the calling user. Stores via the same storage abstraction spec 006 uses (S3-compatible in production, local filesystem in dev). Returns `{ attachmentid, filename, sizebytes }`. The `attachmentid` is included in the next `POST /api/pa/sessions/:id/messages` call as part of `attachmentids`. PDFs up to 10MB upload within 5 seconds. `GET /api/chat/attachments/:id` — authenticated download. Validates the caller is the uploader OR is a member of the session's owning user (per spec 011 auth). Serves the file with `Content-Disposition: attachment` and the original `filename`. **Files are NEVER served via a public URL** — only via this authenticated route. `GET /api/chat/preferences` returns the caller's `userchatpreferences` row with defaults filled in for any unset fields. `PATCH /api/chat/preferences` updates `{ enterkeysends, voiceautosend, layout }` — any subset; takes effect immediately. **Session export passthrough**: spec 005's session-export UI invokes the spec 002-owned `GET /api/pa/sessions/:id/export?format=markdown|pdf|txt` — spec 005 does NOT duplicate that endpoint. Wire the FR-007 UI by adding an **Export** entry to `ChatSessionItem`'s overflow menu (T002) and to `ChatCommandPalette` (T006); both call the spec 002 endpoint and trigger a file download with `Content-Disposition: attachment` filename `{session-name}-{YYYY-MM-DD}.{ext}`. PDF export is the responsibility of spec 002's endpoint (which delegates to spec 006).

### T008 — Studio handoff: detect creation intent + StudioHandoffCard + full-screen expansion + Archie pre-fill + back-to-chat collapse

Implement FR-008. The PA backend (spec 002) is responsible for recognising creation intents (new skill or agent) and returns a response with `actiontype: 'studio_handoff'` alongside the reply text PLUS a `handoffpayload: { type: 'skill' | 'agent', description: string }`. **When the PA response includes `actiontype: 'studio_handoff'`**, `ChatMessageRenderer` (T003) renders a `StudioHandoffCard.tsx` below the PA's response message. The card shows the detected type (`"skill"` or `"agent"`), the description that will be passed forward, and two buttons: **Open Skill Builder** / **Open Agent Builder** (label depends on `handoffpayload.type`) and **Keep chatting**. Clicking **Keep chatting** dismisses the card; the conversation continues normally in the panel. **Clicking Open Skill/Agent Builder**: (1) the `AIChatPanel` transitions to full-screen — animate the panel from `33vw` to `100vw` over 250ms (ease-out); (2) the full spec 004 Studio layout renders inside the now-full-screen panel: `SkillStudio` or `AgentStudio` depending on `handoffpayload.type`; (3) the Studio Assistant (Archie) is initialised with `handoffpayload.description` as if the user had typed it into Archie's input — the community similarity search fires immediately per spec 004 FR-010; (4) all spec 004 FRs apply in full while the Studio is active. The full-screen Studio shows a **← Back to chat** button in the top-left. Clicking Back-to-chat OR pressing Escape (handled by T006 — Escape collapses Studio when Studio is active rather than closing the panel) collapses back to the 33vw panel with a reverse animation. **On Back-to-chat AFTER a skill/agent was saved**: the PA sends a confirmation message into the session thread — `"Skill {name} has been created and is available to use."` or `"Agent template {name} is ready."` — via spec 002. The session context is preserved exactly as it was before the Studio opened (scroll position, draft input, attachment preview bar). **On Back-to-chat AFTER cancelling without saving**: no message is added to the thread; the panel simply resumes its previous state. The `OpenAIChatPanel` action subscriber (T001) is NOT re-triggered by Back-to-chat — the panel was never closed, just visually expanded.
