## Task Details

### T001 — Async job queue infrastructure + S3-compatible file storage with signed-URL downloads + TTL expiry

Implement FR-005 and FR-006 + the foundational data model. Create tables per project DB conventions (lowercase, no separators, UUID v7 PKs, FK names match parent + `id`): `genjobs` (`id` = the `jobid` returned to callers; `userid`, `format` enum `pdf`/`excel`/`pptx`/`image`, `templatekey` nullable, `callbackurl` nullable, `status` enum `queued`/`processing`/`complete`/`error`/`expired`, `fileid` nullable FK → genfiles.id, `errormessage` nullable, `estimatedseconds`, `startedat` nullable, `completedat` nullable, `createdat`, `expiresat = createdat + 1 hour`), `genfiles` (`userid`, `format`, `filename` e.g. `Q2-engineering-2026-05-10.pptx`, `sizebytes`, `storagekey` — internal path NEVER a public URL, `createdat`, `expiresat = createdat + GEN_FILE_TTL_DAYS days`). Generate migrations via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. **Async job queue**: implement an in-process queue processor honoring `GEN_MAX_CONCURRENT_JOBS` (default 5) — when at capacity, new requests queue behind existing ones. Jobs queued but not started within 1 hour move to `status: 'expired'` (worker checks against `expiresat`). `GET /api/gen/jobs/:id` returns `{ jobid, status, format, fileurl?, sizebytes?, errormessage?, estimatedseconds, startedat?, completedat? }`. **Callback delivery**: on completion when `callbackurl` is set, POST `{ jobid, status, fileurl?, errormessage? }` — 3 retry attempts with exponential backoff. **File storage abstraction**: files stored in S3-compatible object storage in production OR local filesystem in development. Config: `GEN_STORAGE_PROVIDER` (`s3` | `local`), `GEN_STORAGE_BUCKET`, `GEN_STORAGE_ENDPOINT`. S3-compatible storage REQUIRED in production; local filesystem is development-only. **All `fileurl` values include a short-lived signed token (24-hour validity)**. Unsigned access to storage is BLOCKED — anonymous access returns 403. `GET /api/gen/files/:id` validates the signed token and serves the file with `Content-Disposition: attachment` so browsers download rather than render. `DELETE /api/gen/files/:id` lets the file owner delete a file before natural expiry. **Daily cleanup job** deletes expired `genfiles` rows AND their backing storage objects atomically (delete object first; on success delete row — fail-forward leaves row pointing to nonexistent storage, which the next cleanup pass will reap). If generation fails, NO partial file is stored — `genjobs.status` is set to `error` and `fileid` remains null.

### T002 — PDF generation via @react-pdf/renderer + system templates (report-standard / invoice-basic / summary-onecolumn)

Implement FR-001. `POST /api/gen/pdf` body `{ template, data, options?, async?, callbackurl? }`. `data` MUST match the template's `dataschema` (JSON Schema validation — invalid data returns HTTP 422 with the validation error path). Files under 1MB returned synchronously with `{ fileurl, filename, sizebytes }` in the response within 5 seconds. Files over 1MB OR with `async: true` queue as `genjobs` and return `{ jobid, status: 'queued', estimatedseconds }` immediately. Use `@react-pdf/renderer` (MIT) — JSX-based with `<Document>`, `<Page>`, `<Text>`, `<Image>` components. **PDFs support**: text with fonts (embed at least Helvetica + Times — additional fonts loaded from `src/assets/fonts/`), tables (rendered via `<View>` with `flexDirection: 'row'`), images (URL or base64), headers/footers (rendered with `fixed` prop), page numbers (via `render` callback receiving `{ pageNumber, totalPages }`), multi-page layout. **Fallback for complex PDF layouts**: `puppeteer` (HTML-to-PDF) — heavier but supports any CSS layout. Wire `puppeteer` as a secondary engine selectable per template via `gentemplates.metadata.engine`. **System templates bundled** (read-only, prefixed `system:`): `report-standard` (cover page + section headings + body text + page numbers), `invoice-basic` (sender/receiver block + line-item table + totals row + payment instructions), `summary-onecolumn` (single-column layout for AI-generated summaries with title, body, optional image). Seed these into `gentemplates` on first launch with `issystem: true`. Each PDF generation writes a `genlog` row (T007).

### T003 — Excel generation via exceljs + system templates (data-export-standard / financial-summary)

Implement FR-002. `POST /api/gen/excel` body `{ template?, sheets: [{ name, columns: [{ key, header, width? }], rows: [...] }], options?, async?, callbackurl? }`. Without `template`, the engine applies default styling: bold header row, alternating row fill, auto-fit column widths. Synchronous for up to 10,000 rows total across all sheets (target ≤ 10 seconds); async for larger. Use `exceljs` (MIT) — full XLSX authoring with styles, formulas, charts. **Supports**: multiple sheets, cell formatting (numbers / dates / currency via `numFmt`), formulas (`{ formula: 'SUM(A2:A10)' }`), freeze top row (`sheet.views = [{ state: 'frozen', ySplit: 1 }]`), auto-filter on header row (`sheet.autoFilter = { from: 'A1', to: <last col><last row> }`), conditional formatting (per-cell `fill`/`font`), column widths auto-fit or per-column `width` override. **System templates** (seeded as `issystem: true`): `data-export-standard` (bold header row + alternating fill + auto-filter — used for any generic tabular export), `financial-summary` (heading rows + currency formatting + bold totals row + summary chart). The `dataschema` validates `sheets[].columns` and `sheets[].rows` shape against the template's expectations. Each Excel generation writes a `genlog` row.

### T004 — PowerPoint generation via pptxgenjs + system templates (pitch-deck / status-update / technical-report)

Implement FR-003. `POST /api/gen/presentation` body `{ template?, slides: [{ title, content, layout?, imageurl? }], options?, async?, callbackurl? }`. Each slide's `content` can be: text with inline formatting (bold/italic via lightweight markdown subset), bullet lists (array of strings), table data (`{ headers, rows }`), or an image (URL — fetched server-side and embedded). Synchronous for 1–5 slides (target ≤ 15 seconds); async for > 5 slides (job-queued, callback on completion). Use `pptxgenjs` (MIT) — programmatic slide generation. **System templates** (seeded `issystem: true`): `pitch-deck` (cover + agenda + content slides + closing — wide colour band + sans-serif font, large title placement), `status-update` (compact layout with title block + 2-column body — small footer with date + page number), `technical-report` (clean layout with title bar + body area + sidebar callout box). Each template defines: background (solid colour or master image), fonts (heading + body face + size), title placement (master slide), content area (placeholder rect), logo position (top-right or bottom-left, configurable via template metadata). Per-slide `layout` override picks among the template's defined master slides. Each PPTX generation writes a `genlog` row.

### T005 — AI image text-to-image generation (FR-004): provider abstraction + DALL-E 3 default + shared daily limit

Implement FR-004. `POST /api/gen/image` body `{ prompt, size?, quality?, style? }`. Defaults: `size: '1024x1024'`, `quality: 'standard'`. **Always synchronous** — returns `{ fileurl, filename, prompt, model, generatedat }` within 15 seconds. **Provider abstraction**: define a `GenImageProvider` interface with `generate(prompt, opts) → Promise<{ buffer, model, contentType }>` AND a `supportsImageEdit: boolean` capability flag (consumed by T006). Default provider: OpenAI DALL-E 3 via the `openai` npm package — `client.images.generate({ model: 'dall-e-3', prompt, size, quality, response_format: 'b64_json' })`. Configurable via env vars: `GEN_IMAGE_PROVIDER` (default `openai`) and `GEN_IMAGE_API_KEY`. Additional providers can register at startup; each declares its `supportsImageEdit` flag. **Per-user daily limit**: 10 images per user per day, configurable via `GEN_IMAGE_DAILY_LIMIT`. Implement the limit as a sliding window — counts every successful `POST /api/gen/image` AND `POST /api/gen/image-edit` (T006) call across the past 24 hours per `userid`. Excess returns HTTP 429 `{ error: 'daily_limit_reached' }`. **Prompt policy violation** from the provider → HTTP 422 `{ error: 'prompt_rejected', reason }` — no file created. **If no image provider is configured**: HTTP 503 `{ error: 'image_generation_not_configured' }` — NEVER return a placeholder fallback. Store the returned image bytes in `genfiles` per T001's storage path; `filename` is `gen-image-{shortid}.png`. Each image generation writes a `genlog` row with `format: 'image'`, `prompttruncated` (first 200 chars only — full prompts NEVER persisted), `sizebytes`, `latencyms`.

### T006 — AI image edit / image-to-image (FR-009): multipart endpoint + provider capability check + transparent backgrounds + spec 007 integration

Implement FR-009. `POST /api/gen/image-edit` accepts multipart upload: `image` (source — JPG/PNG/WebP, ≤ 10MB), `prompt` (text instruction), optional `size` (default `1024x1024`), `background` (`opaque` | `transparent`, default `opaque`), `mask?` (optional inpainting mask image). **Provider capability check**: each registered image provider (T005) declares `supportsImageEdit: boolean`. The default OpenAI provider routes to `POST /v1/images/edits` with `model: 'gpt-image-1'`. **If `GEN_IMAGE_PROVIDER` is configured to a provider with `supportsImageEdit: false`** (e.g., a stable-diffusion-only deployment that hasn't enabled img2img), the engine returns HTTP 501 `{ error: 'image_edit_not_supported_by_provider', provider }` — NO file created, NO silent fallback. **If no image provider is configured at all**: HTTP 503 `{ error: 'image_generation_not_configured' }` — same error as FR-004. **Sync/async split**: synchronous when source image ≤ 2MB — returns `{ fileurl, filename, prompt, model, generatedat }` within 30 seconds. Asynchronous for source > 2MB — returns `{ jobid, status: 'queued', estimatedseconds }` and posts to `callbackurl` on completion via the T001 job mechanism. Prompt policy violation → HTTP 422 `{ error: 'prompt_rejected', reason }` — no file created. **Shared daily limit**: image-edit calls count against the SAME `GEN_IMAGE_DAILY_LIMIT` bucket as FR-004 text-to-image. When exceeded → HTTP 429 `{ error: 'daily_limit_reached' }`. There is NO separate cap for edits. **Source image is buffered in memory only** — never written to long-term storage. The output file is stored normally per T001 (signed URL, expiry, etc.). **Spec 007 integration**: when called from spec 007's avatar conversion flow, the end-user's `userid` (NOT the calling service identity) MUST be forwarded in the request and recorded in `genlog` so the daily-limit bucket attribution is correct. Each image-edit invocation writes a `genlog` row exactly like FR-004 — `format: 'image'`, `prompttruncated` (first 200 chars of `prompt`), `sizebytes`, `latencyms`. **The source image is NEVER logged**.

### T007 — Document template management (FR-007) + generation log (FR-008) + admin-only template CRUD

Implement FR-007 and FR-008. Create `gentemplates` table: `key` (UNIQUE globally — system keys prefixed `system:` are globally reserved), `format` enum `pdf`/`excel`/`pptx`, `name`, `description`, `dataschema` (jsonb — JSON Schema callers must satisfy), `templatefilekey` (storage path of the template file), `issystem` boolean (true for bundled system templates — read-only), `createdat`, `updatedat`. Create `genlog` table: `userid`, `format`, `templatekey` nullable, `prompttruncated` nullable (first 200 chars only — image prompts only), `jobid` nullable, `status` (final), `sizebytes` nullable, `latencyms`, `createdat`, `expiresat = createdat + 90 days`. Generate migrations; validate. **Template APIs**: `GET /api/gen/templates` returns all templates (system + org-specific) with key, format, name, description, and `dataschema` — visible to any authenticated user (callers need to discover schemas). `GET /api/gen/templates/:key` returns template detail. `POST /api/gen/templates` (admin only — `hasSystemFlag('issystemadmin')` OR org admin via spec 011) creates a custom template: `{ key, format, name, description, dataschema, templatefile (multipart upload) }`. Validates: `key` is unique per org AND not starting with `system:` (HTTP 422 with `{ error: 'reserved_key_prefix' }`); `format` is one of `pdf`/`excel`/`pptx`; `dataschema` is a valid JSON Schema. Template usable by callers within 60 seconds of saving (no caching delay). `PATCH /api/gen/templates/:key` (admin only) updates a custom template. **System templates are read-only** — `PATCH` or `DELETE` attempts return HTTP 403 `{ error: 'system_template_readonly' }`. `DELETE /api/gen/templates/:key` (admin only) removes a custom template; system templates cannot be deleted. **Generation log**: every `POST /api/gen/*` request creates a `genlog` row (called by every generation task — T002-T006). `GET /api/gen/log` (admin only) returns paginated log entries filterable by `userid`, `format`, `status`, date range. **Log rows purged after 90 days** by the daily cleanup job (same job that reaps expired `genfiles` and `genjobs`). **Image prompts stored in `genlog` are truncated to 200 characters** — full prompts NEVER persisted.

### T008 — Frontend: GeneratedFileCard / GenerationProgressCard / GeneratedImageCard + GenTemplateManager admin UI

Build the shared result components and admin template-management UI under `src/ui/components/gen/`. These are used by spec 002 (PA), spec 004 (Studio), and spec 005 (AI Chat) wherever files are delivered. **`GeneratedFileCard.tsx`** — rendered in AI Chat and PA channel when a file is ready. Shows: file-type icon (PDF / XLSX / PPTX badge with colour), `filename`, formatted `sizebytes` (e.g., `2.3 MB`), Download button (opens the signed `fileurl` in a new tab — `Content-Disposition: attachment` makes the browser save rather than render), expiry note (`"Link expires in N days"` calculated from `expiresat`). On click of an in-card overflow menu: **Delete** (calls `DELETE /api/gen/files/:id`, removes the card on success). **`GenerationProgressCard.tsx`** — shown during async jobs while `status === 'queued'` or `'processing'`. Shows: format icon, headline `"Generating your {PDF|Excel|presentation}…"`, estimated time remaining (from `estimatedseconds` + ticker), progress bar (indeterminate during `queued`; determinate during `processing` using `startedat` against `estimatedseconds`). Polls `GET /api/gen/jobs/:id` every 3 seconds. On `status === 'complete'` transitions to `GeneratedFileCard` (replaces the progress card in place — same DOM node so animations remain smooth). On `status === 'error'` shows the `errormessage` with a Retry action that re-POSTs the original request. On `status === 'expired'` shows `"Generation job expired — try again."` **`GeneratedImageCard.tsx`** — inline image result. Shows: preview thumbnail (`<img src=fileurl>` lazy-loaded), prompt text truncated to ~120 chars with `...` and click-to-expand, model label (`DALL-E 3` etc.), full-size view button (opens lightbox modal with the original-resolution image), Download button. **Admin UI**: `GenTemplateManager.tsx` — admin page listing all templates with format badge (PDF / XLSX / PPTX) and system/custom label. System templates rendered with a lock icon and disabled edit/delete actions. Create, edit, delete actions for custom templates only. `GenTemplateForm.tsx` — create/edit form: `key` field (validated against the reserved `system:` prefix client-side), `format` dropdown, `name`, `description`, JSON Schema editor for `dataschema` (a `<textarea>` with `application/schema+json` Monaco mode + a side-pane preview of the schema's expected shape rendered as a labeled tree), template file upload (drag-and-drop). Save fires `POST` or `PATCH /api/gen/templates`; success shows a toast and returns to the manager list. Both admin components are accessible only to callers passing spec 011's `requireAuth({ orgRole: 'admin' })` OR `requireAuth({ systemFlag: 'issystemadmin' })`.
