## Task Details

### T001 — Push token registry: pushtokens / notificationpreferences / pushdeliveries / pushtaps tables + token-registration APIs

Implement FR-001 + data model for the entire spec. Create tables per project DB conventions (lowercase, no separators, UUID v7 PKs, FK names match parent + `id`): `pushtokens` (UNIQUE on `(userid, deviceid, platform)`, columns `userid`, `deviceid`, `platform` enum `ios`/`android`/`web`, `token`, `webpushp256dh` nullable, `webpushauth` nullable, `apnsenv` nullable `development`/`production`, `appversion`, `locale`, `lastusedat`, `lastfailedat`, `failurecount`, `createdat`; index on `userid` — primary fanout lookup), `notificationpreferences` (UNIQUE userid, `pushenabled` boolean default true, `quiethoursstart` time nullable, `quiethoursend` time nullable, `quiethourstimezone` IANA text, `pertypeprefs` jsonb, `perchannelprefs` jsonb, `updatedat`), `pushdeliveries` (30-day retention via `expiresat`, FK `tokenid` → pushtokens, columns `eventtype`, `eventid` nullable FK → spec 003 wsevents, `payloadkind` enum `silent`/`visible`/`fallback-visible`, `provider` enum `apns`/`fcm`/`webpush`, `status` enum `sent`/`delivered`/`failed`, `errormessage`, `latencyms`, `sentat`, `deliveredat`; indexes on `(userid, sentat DESC)` and `(eventid)` for fallback-timer lookup), `pushtaps` (90-day retention, FK → pushdeliveries, pseudonymous). Generate migrations via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. Implement endpoints: `POST /api/push/tokens` body `{ deviceid, platform, token, webpushp256dh?, webpushauth?, apnsenv?, appversion, locale }` — auth-required (spec 011); re-registration with the same `(userid, deviceid, platform)` triple UPDATES the existing row's `token`, NOT inserts a new row. `DELETE /api/push/tokens` body `{ deviceid }` hard-deletes all tokens for the calling user on that device — invoked by spec 011 FR-007 logout in the same atomic sequence as `SecureKeyStore.clearUser`. `GET /api/push/tokens` lists caller's registered tokens with `token` redacted to last 6 chars (surfaces in `RegisteredDevicesList` UI). When a provider returns `unregistered` / `BadDeviceToken` / `410 Gone`, the fanout worker hard-deletes the offending row in-line — no separate cleanup job for these. A daily cleanup job prunes tokens with `lastusedat` older than 60 days and no successful delivery. Token values stored as-is (opaque device-specific strings, not credentials); NEVER logged in plaintext beyond last 6 chars.

### T002 — Foreground-aware fanout worker: WSBroker subscriber + lifecycle ping + push-vs-no-push decision matrix

Implement FR-002. The fanout worker subscribes to spec 003 `WSBroker.publish` stream as a downstream subscriber — push fanout is NOT in the publish path; a push failure NEVER prevents WS delivery. For each published event, the worker queries spec 003 `wssessions` for active sessions on the affected `user:{id}` scope. **Decision matrix**: foreground (any platform) with WS connected → no push (WS handles delivery); foreground with WS just dropped → no push (WS reconnects within 30s); mobile background or app closed with no WS → push; browser tab hidden with idle WS alive → no push (JS still receives events); browser closed → push via Web Push. A session is considered foreground if `lastheartbeat` is within 60 seconds AND the device's client has reported foreground state in the last 2 minutes. **Multi-device**: each device is evaluated independently — if user has phone (background) + laptop (foreground WS), phone gets a push. Implement `POST /api/push/lifecycle` body `{ deviceid, state: 'foreground' | 'background' }` — client lifecycle ping invoked by spec 008 `MobileLifecycle` on every foreground/background transition. The server caches lifecycle state in memory (or Redis when running clustered) for the 2-minute decision window. Worker latency target: `WSBroker.publish` → push provider call ≤ 500ms p95 under normal load. In v1, the worker runs in-process with `LocalBroker`; when spec 003 switches to `RedisBroker` for horizontal scale, the worker can move to its own host subscribed to the same Redis pub/sub channel.

### T003 — Event-to-payload mapping registry: src/server/push/event-mappings.ts with full default mapping table

Implement FR-003. Create the registry `src/server/push/event-mappings.ts` — one entry per WSEvent type. Each entry exports: `eventType: string`, `shouldPush(event, userPrefs) → boolean`, `buildPayload(event, userLocale) → PushPayload`, `bypassQuietHours: boolean`, `throttle?: { perWindowMs, maxPerWindow }`, `deepLink: (event) → string`. Adding a new event type without a mapping defaults to no-push (fail-safe). **Default mapping table**: `pa.notification` briefing → visible, `"Your morning briefing is ready"`, link `yappchat://pa/notification/:id`. `pa.notification` calendar_reminder → visible, `bypassQuietHours: true`, `"{title} starts in 15 min"`, link `yappchat://video/:roomid` if the meeting has a YappChat video room else `yappchat://pa/notification/:id`. `pa.notification` project_overdue → visible, `"{taskname} is overdue"`, link `yappchat://pa/notification/:id`. `pa.notification` email_alert → visible, `"Email from {sender}"`. `pa.notification` pending_messages → throttled 1 per 30 min, `"{count} new messages"`. `message.inbound` with mention of user → visible (silent-then-fetch for E2E), `"{sender} mentioned you in {channel}"`, link `yappchat://conversation/:id?messageid=:mid`. `message.inbound` direct message to user → visible (silent-then-fetch for E2E), `"{sender}: {preview}"`. `message.inbound` channel-no-mention → per-channel preference (default off), `"New message in {channel}"`, link `yappchat://conversation/:id`. NEVER push: `message.delivery_status`, `message.typing_*`, `presence.*`, `videoroom.ended`, `subagent.status: working`/`spawned`/`waiting_for_input`, `skill.update_available`, `mcp.server_status`, `directory.*`, `channel.health`. `videoroom.participant_joined` → push only if user is invitee not yet joined, `"{name} is in the call"`, link `yappchat://video/:id`. `subagent.status: completed` → visible, `"Your subagent finished"`, link `yappchat://session/:parentsessionid`. `subagent.status: error` → visible, `"Your subagent failed"`. `agent.message` (spec 001 FR-010 dev agents) → per-user opt-in (default off), `"{agent}: needs your input"`, link `yappchat://agent/:id`. Throttle state is per-user-per-mapping, in-memory or Redis, on a sliding window. Localised body strings are loaded from a resource bundle keyed by user locale (from `mobiledevices.locale` or browser `Accept-Language`). English is the v1 baseline; the system supports additional locales without code changes.

### T004 — APNs (iOS) provider adapter with token-auth + sandbox/production environment routing

Implement FR-004. Config (server secrets, never committed): `APNS_AUTH_KEY` (.p8 file path or content), `APNS_KEY_ID`, `APNS_TEAM_ID`, `APNS_BUNDLE_ID` (matches spec 008 `com.wxperts.yappchat`), `APNS_ENVIRONMENT` (`development` | `production`). Use `@parse/node-apn` (or equivalent) over HTTP/2 with token-based authentication. Connection pooling with HTTP/2 keep-alive; reconnect on idle drop. **Visible payload**: `{ aps: { alert: { title, body }, sound: 'default', 'thread-id': <conversationid|sessionid|'pa'> }, link: <deeplink>, type: <eventType> }`. **Silent payload (E2E wake-and-fetch)**: `{ aps: { 'content-available': 1 }, type: 'wake-and-fetch', scope, eventid }` — no `alert`, no `sound`. iOS delivers silently; the app's `BackgroundFetch` handler performs the fetch + local notification (FR-006). **Per-environment routing**: `APNS_ENVIRONMENT=development` targets `api.sandbox.push.apple.com`; `production` targets `api.push.apple.com`. EAS preview builds use development; EAS production builds use production. The shell tells the server which environment via the `apnsenv` field on `pushtokens` registration (per-token, not per-deployment — different builds on the same deployment may target different APNs environments). **Response handling**: 200 → success, log delivery latency; 410 → delete token from `pushtokens` in-line; 429 → exponential backoff + retry up to 3 times; 5xx → retry with backoff, then give up after 3 attempts. Apple's silent-push throttling is acknowledged — the spec 002 PA monitoring loop produces many silent pushes per user per day. The FR-006 fallback path handles throttle rejections gracefully.

### T005 — FCM (Android) provider adapter with notification channels + doze guidance

Implement FR-005. Config (server secrets): `FCM_SERVICE_ACCOUNT_JSON` (path or content), `FCM_PROJECT_ID`. The service account has `roles/firebasenotifications.admin` role ONLY — no broader Firebase permissions. Use `firebase-admin` SDK's `messaging.send()`. One Firebase project for all environments — differentiate by topic prefix or token registration source, NOT separate Firebase projects. **Visible payload**: `{ token, notification: { title, body }, data: { link, type, eventid }, android: { priority: 'high', notification: { channel_id: 'default' } } }`. **Silent payload (data-only for E2E wake-and-fetch)**: `{ token, data: { type: 'wake-and-fetch', scope, eventid, link }, android: { priority: 'high' } }` — no `notification` block; the app's data-handler renders a local notification with decrypted content. **Android notification channels** (FCM Android 8+ requirement) registered by the mobile shell on first launch: `default` (general), `urgent` (calendar, mentions — bypass DND), `quiet` (low-priority message previews). **Battery whitelisting / OEM doze**: Xiaomi, Samsung, OPPO, etc. implement aggressive battery management beyond stock Android — even high-priority FCM pushes can be delayed for 10+ minutes. Spec 008 onboarding already includes "disable battery optimisation for YappChat" guidance. This adapter detects missing client-side ack within 60s of a high-priority push and emits a ONE-TIME PA notification (`bypassQuietHours: true`) with that guidance. Dedup so this fires at most once per user per device per 7 days. **Response handling**: `messaging/registration-token-not-registered` → delete from `pushtokens`; `messaging/quota-exceeded` → backoff and retry; `messaging/server-unavailable` → retry 3 times then give up.

### T006 — Web Push provider adapter: VAPID auth + service worker + browser subscription flow

Implement FR-009. Generate VAPID keys once per environment, store in server secrets: `WEBPUSH_VAPID_PUBLIC_KEY`, `WEBPUSH_VAPID_PRIVATE_KEY`, `WEBPUSH_VAPID_SUBJECT`. Expose `GET /api/push/webpush/vapid-public-key` so the client can subscribe (public key only — never the private key). **Browser flow**: client requests notification permission, calls `serviceWorker.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey })`, POSTs the resulting `PushSubscription` (`endpoint`, `keys.p256dh`, `keys.auth`) to `POST /api/push/tokens` with `platform: 'web'`. Implement the service worker at `apps/web/public/sw.js` — it handles two events: `push` (parses the encrypted payload, renders `self.registration.showNotification(title, options)` with the deep link in `data.link`) and `notificationclick` (calls `clients.openWindow(link)` to focus or open a tab). **Server-side**: use the `web-push` npm package — `webpush.sendNotification(subscription, payload, { vapidDetails })`. Payload is encrypted client-side per the Web Push Protocol. **Web Push always uses visible payloads** — silent web push exists but is poorly supported across browsers; the complexity isn't worth it in v1. E2E content for web is fetched at the moment the user clicks the notification. **Payload size cap**: ~4KB across browsers — the fanout worker MUST truncate payloads if needed and include a deep link to fetch the full content on click. **Response handling**: 410 or 404 from the push endpoint → delete the row from `pushtokens` (subscription expired). The service worker MUST NOT contain auth credentials, decryption keys, or business logic — its only job is to render the notification and route the click. Targets Chrome, Edge, Firefox, Safari 16+.

### T007 — Silent-push-then-fetch flow + 30s visible-push fallback timer + ack endpoint + delivery analytics + rate limiting

Implement FR-006, FR-010, FR-011. **Silent-push-then-fetch (FR-006)**: for events where the source message has `encryptiontype: 'e2e'` or `'agent-e2e'`, the worker FIRST sends a silent payload containing `{ type: 'wake-and-fetch', scope, eventid, link }` and NO plaintext. Push providers never see plaintext content for E2E messages. The mobile shell's notification handler (T008) processes wake-and-fetch payloads. **Server-side fallback timer**: 30 seconds after a silent push is sent, if no `POST /api/push/ack` has arrived for that `eventid`, the worker sends a generic visible push — title `"YappChat"`, body `"You have a new message"`, deep link to the relevant scope. NEVER include sender names, channel names, or content in the fallback body for E2E messages. Use the `pushdeliveries (eventid)` index to find the matching delivery row. Non-E2E messages (`encryptiontype: 'platform'` — Slack, Discord, etc.) skip the silent path and go straight to visible push with the platform-provided plaintext content. **Ack endpoint (FR-010)**: `POST /api/push/ack` body `{ eventid, deviceid, deliveredat }` — called by the mobile shell once a silent push has been received and the fetch + decrypt have completed. Updates the matching `pushdeliveries` row to `status: 'delivered'` with `deliveredat`. Also implement `POST /api/push/tap` body `{ pushdeliveryid }` — writes a `pushtaps` row for engagement analytics. **Analytics**: log a `pushdeliveries` row on every send with `latencyms` measured from `WSBroker.publish` → provider call return. `GET /api/push/stats` (admin only — system admin per spec 011) returns aggregate stats: send counts by provider, delivery success rate, p50 / p95 latency, fallback rate, taps per event type. Provider error rates above thresholds (> 5% APNs `BadDeviceToken`, > 10% FCM unavailable) trigger a PA notification to `andy@wxperts.com` — alert dedup 1 per 24h per provider. Per-user delivery history is NOT exposed via API; operators access individual debug records via direct DB query under explicit support workflow. **Rate limits (FR-011)**: per-event-type throttle (mapping's `throttle` field) applied first. Per-user push budget: max 60 pushes per hour per user — excess is dropped (not queued) with a warn log. If a user hits the per-user budget more than 3 times in 24h, auto-pause pushes for that user for 1 hour and post a PA channel notification: `"Pushes paused due to unusually high volume — investigate"`. Per-deployment global cap: max 10,000 pushes per minute across all users — soft drop, WS continues. All cap and budget values are env-tunable.

### T008 — Client handlers (PushHandler, web subscription, service worker) + NotificationSettingsScreen + preferences API

Implement FR-007 (notification preferences) + FR-008 (tap-target dispatch) + the small client surface. **Preferences API**: `GET /api/push/preferences` returns the caller's `notificationpreferences` row with defaults filled in for any unset fields. `PATCH /api/push/preferences` updates any subset of fields (master `pushenabled`, `quiethoursstart`, `quiethoursend`, `quiethourstimezone`, `pertypeprefs`, `perchannelprefs`); takes effect on the next event. Quiet hours timezone defaults to the most recently registered device's `mobiledevices.timezone`; the user can override in settings. Per-channel preference values: `'all'` (push every message), `'mentions'` (only when user mentioned — default for shared channels), `'off'` (never push for this channel — DM channels stay on by default). Quiet hours are respected by the fanout worker; the per-mapping `bypassQuietHours: true` flag overrides for calendar reminders only. **Quiet-hours-suppressed pushes are NEVER replayed when quiet hours end** — events remain readable via WS replay or REST refresh; replay would create a morning flood. `POST /api/push/test` sends a sample push to all of the caller's registered tokens — used to verify a new device is correctly wired. **Mobile client handler**: `apps/mobile/src/PushHandler.tsx` — mounts inside spec 008 `MobileRoot`. Subscribes to `expo-notifications` events. Handles foreground notifications, taps, and silent `wake-and-fetch` payloads — opens authenticated WS, fetches via `GET /api/engine/messages/:id` (or `GET /api/pa/sessions/:id/messages?since=...`), decrypts client-side using spec-001 key, then calls `Notifications.scheduleNotificationAsync` to render a local visible notification with the decrypted preview. If fetch + decrypt completes within 25 seconds of the silent push arriving, ONE notification is rendered (the local one), not two. On `POST /api/push/ack` success, the fallback timer is satisfied and no generic visible push is sent. Dispatches taps to spec 008 `DeepLinkRouter`. **Web client**: `apps/web/src/push-subscription.ts` — subscribes the browser to Web Push when permission is granted, posts the subscription to `/api/push/tokens`. **Service worker** at `apps/web/public/sw.js` (referenced from T006 — implementation lives there). **Tap-target dispatch (FR-008)**: every push payload includes a `link` field of the form `yappchat://...` (mobile) or `https://yappchat.app/...` (web universal link). Mobile cold start: spec 008 `DeepLinkRouter` reads `Linking.getInitialURL()` and navigates before first render. When the user taps a notification while the app is open, the in-app handler reads the link and dispatches to `DeepLinkRouter` directly without round-tripping through the OS. Each tap calls `POST /api/push/tap`. If the deep-linked resource no longer exists (deleted message, ended video room), the target surface shows a friendly empty state — the app does NOT crash or render an error toast. **Settings UI** under `packages/ui/src/notifications/` (shared web + mobile via spec 008 FR-002): `NotificationSettingsScreen` (master switch, quiet hours picker, per-type toggle grid, per-channel preferences list, "Test push" button, registered devices list), `QuietHoursPicker` (time-range picker with IANA timezone select), `PerTypePreferencesGrid` (toggle grid for each event type with a friendly label + description of when it fires), `PerChannelPreferencesList` (all / mentions / off picker per channel), `RegisteredDevicesList` (shows where YappChat is installed and receiving pushes; each row has a "Sign out from this device" action that calls spec 011 logout + `DELETE /api/push/tokens`). Denied-permission UX matches spec 008 FR-005 — surface a "You denied notifications — open Settings to enable" path.
