## Task Details

### T001 ‚Äî WebSocket server + connection lifecycle + auth token validation + wssessions registry

Implement FR-001. Create `wssessions` table per project DB conventions (UUID v7 PK ‚Äî also serves as the `sessionid` sent to the client on connect): `userid` (authenticated user), `subscriptions` text[] (current scope subscriptions for this session ‚Äî see T002), `connectedat`, `lastheartbeat` (updated on each heartbeat ack ‚Äî see T004). Rows are deleted on connection close OR by dead-connection cleanup (T004). Generate migration via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. **WebSocket server** runs on `wss://{host}/ws` ‚Äî TLS required in production; `ws://` permitted ONLY on localhost in development (enforce via env check). Built with the `ws` npm package already present in the workspace at `packages/openclaw/node_modules/ws`. Build on existing OpenClaw gateway WebSocket infrastructure at `packages/openclaw/src/gateway/client.ts` and `packages/openclaw/src/gateway/server-broadcast.ts`. **Auth token validation on connect** (spec 011): client connects by passing a short-lived auth token ‚Äî either `wss://host/ws?token=<token>` OR `Authorization: Bearer <token>` upgrade header. Token validated BEFORE any data is exchanged. Invalid or expired token ‚Üí close with code `4001` immediately, no further messages sent. Token validation MUST happen before subscriptions are accepted ‚Äî client trust is NEVER assumed. **On successful connect**, server sends a `connected` event as the FIRST message: `{ sessionid, userid, servertime }`. Insert `wssessions` row. **Multiple concurrent sessions per user supported** (multiple browser tabs, mobile + desktop) ‚Äî all receive events that match their subscriptions independently. **Capacity target**: at least 1,000 concurrent WebSocket connections on a single Node.js process (event delivery latency NOT exceeding 200ms at this scale). **Token expiry mid-session**: if a client's auth token expires WHILE connected, the WebSocket connection remains open ‚Äî token is only validated on connect. Rotating credentials should NOT drop active sessions. Admin endpoints: `GET /api/ws/sessions` (admin only ‚Äî list active sessions with userid, subscriptions, connected time), `DELETE /api/ws/sessions/:id` (admin only ‚Äî force-close a specific session, e.g. when an account is deactivated).

### T002 ‚Äî Subscription management with scope authorization (user/channel/org/agent/videoroom/pairing/broadcast)

Implement FR-002. **Client ‚Üí server messages**: `{ type: 'subscribe', scopes: [...] }` subscribes to one or more scopes in a single message; `{ type: 'unsubscribe', scopes: [...] }` removes subscriptions (e.g., when closing a conversation tab). Subscriptions are stored in memory per session (the `wssessions.subscriptions` array is the cached projection ‚Äî primary storage is in-process Map keyed by sessionid). NOT persisted across reconnect ‚Äî on reconnect the client re-subscribes as part of the resume handshake (T005). **Scope authorization rules** enforced server-side (NEVER trust the client): `user:{userid}` ‚Äî caller's authenticated `userid` must match the scope's userid; `channel:{channelid}` ‚Äî caller must be a member of the channel per spec 001 channel membership; `org:{orgid}` ‚Äî caller must be a member of the org per spec 011 `orgmemberships`; `agent:{agentid}` ‚Äî caller must have access to the agent (org admin OR the agent's registrar); `videoroom:{roomid}` ‚Äî caller must be an invitee or a current participant; `pairing:{pairingid}` ‚Äî caller is the user that initiated the pairing AND the pairing nonce in the subscribe message matches `keypairings.nonce` AND `keypairings.expiresat > now()` AND the pairing has NOT yet been completed (subscribe to a `completed` pairing is rejected ‚Äî single-use). Spec 003 enforces by querying `keypairings` at subscribe time. `broadcast` ‚Äî accepted from any authenticated caller. **Unauthorized subscription attempts** are rejected with `{ type: 'error', code: 'unauthorized_scope', scope }` but the connection STAYS OPEN ‚Äî only that specific subscribe is dropped. Valid scopes in the same message are still applied. **Subscription updates** are reflected in the session's in-memory subscription set immediately and used by T003's event router.

### T003 ‚Äî Event envelope + WSBroker abstraction (LocalBroker default, RedisBroker designed-in) + event routing pipeline

Implement FR-003 and the FR-007 broker abstraction. Define the canonical `WSEvent` TypeScript interface in `src/lib/ws-client.ts`: `{ id: string (UUID v7 ‚Äî monotonic, suitable for replay cursor), type: string, scope: string, payload: unknown, ts: number (Unix ms) }`. **Internal publish API**: `wsEngine.publish(event: WSEvent)` is a fire-and-forget call from publishers. NOT a network call ‚Äî runs in-process (LocalBroker) OR via Redis pub/sub (RedisBroker). The internal interface is abstracted behind a `WSBroker` interface with TWO implementations: **`LocalBroker`** (in-process, Map-based, used in v1 ‚Äî default) and **`RedisBroker`** (pub/sub via `ioredis`, used when `WS_BROKER=redis` env var is set). Switching from LocalBroker to RedisBroker requires ONLY setting `WS_BROKER=redis` and `REDIS_URL` ‚Äî NO application code changes. **Event routing**: every published event is routed to all sessions with a subscription matching the event's `scope`. Events delivered to matched clients within 100ms of being published under normal load (single-server deployment) ‚Äî within 200ms at 1,000 concurrent connections (T001's capacity target). Events that cannot be delivered to a momentarily-disconnected client are written to `wsevents` (T005) for replay on reconnect. **Event type catalogue** (all of these are routed via this pipeline ‚Äî each event's scope determines which sessions receive it; the engine has NO knowledge of what the events MEAN, only how to route them): `message.inbound` / `message.delivery_status` / `message.typing_start` / `message.typing_stop` / `presence.online` / `presence.offline` / `presence.in_call` / `channel.health` / `agent.status` / `agent.message` / `pa.notification` / `pa.status` / `directory.member_updated` / `directory.member_added` / `directory.member_removed` / `directory.status` / `videoroom.participant_joined` / `videoroom.participant_left` / `videoroom.ended` / `skill.update_available` / `mcp.server_status` / `subagent.status` / `avatar.conversion_complete` / `keybackup.recovery_alert` / `keypairing.delivered`. Each is owned by its publishing spec (see Notes in source spec for ownership mapping). Server ‚Üí client message types delivered through the pipeline: `event` (wrapped WSEvent ‚Äî primary delivery), plus envelope control messages from T001/T004/T005 (`connected`, `heartbeat_ack`, `replay_start`, `replay_end`, `replay_unavailable`, `error`). **Engine MUST NOT contain business logic** ‚Äî its only job is routing events from publishers to subscribers. Business logic stays in the originating spec's service layer. Event IDs MUST be UUID v7 ‚Äî monotonic ordering is required for the replay cursor in T005.

### T004 ‚Äî Heartbeat + ping-pong + dead-connection detection + client-level heartbeat

Implement FR-004. **WebSocket-level ping/pong**: server sends a WebSocket `ping` frame to each client every 30 seconds. If a client does not respond with a `pong` within 10 seconds, the connection is considered dead ‚Äî closed with code `1001` (Going Away) and the `wssessions` row is deleted. Cleanup is non-blocking ‚Äî dead-connection detection runs on a timer, NOT in the event loop's hot path. **Application-level heartbeat** (complementary to ping/pong): client sends `{ type: 'heartbeat' }` every 25 seconds. The server responds with `{ type: 'heartbeat_ack' }` and updates `wssessions.lastheartbeat` to `now()`. If the client does NOT receive a `heartbeat_ack` within 10 seconds of sending, it treats the connection as dead and begins reconnection (T008). The application-level heartbeat is necessary in addition to ping/pong because some intermediaries (NATs, corporate proxies) silently drop idle TCP connections without sending FIN ‚Äî application-level keepalive catches this faster than ping/pong alone. **Dead-connection cleanup** on disconnect: `wssessions` row deleted, in-memory subscription set freed, presence state updated via T006 (if this was the user's last active session, publish `presence.offline`). The session ID is NOT reused ‚Äî new connections always get a fresh UUID v7 sessionid.

### T005 ‚Äî Event log for replay on reconnect: wsevents table + 5-minute TTL + resume handshake + cleanup job

Implement FR-005. Create `wsevents` table: `id` (UUID v7 PK ‚Äî monotonic, used as replay cursor), `type` (event type), `scope` (routing scope e.g. `channel:abc123`), `payload` jsonb, `ts` bigint (Unix ms), `expiresat` timestamptz (`now() + 5 minutes` at insert ‚Äî INDEXED for cleanup job). Indexes: `(scope, id)` for replay queries filtered by scope and cursor; `(expiresat)` for the cleanup job. Generate migration; validate. **Every published event is inserted** into `wsevents` with a 5-minute TTL ‚Äî happens as part of T003's publish pipeline. Events older than `expiresat` are deleted by a background cleanup job that runs every 60 seconds. **The 5-minute TTL is fixed and intentional** ‚Äî the log is for TRANSIENT recovery, not a message store. Spec 001 owns durable message persistence. **Resume handshake on reconnect** (called by T008's WSClient): client sends `{ type: 'resume', lastEventId: '<uuid>' }`. The engine queries `wsevents WHERE id > lastEventId AND expiresat > now() AND scope IN (client_subscriptions)` (subscriptions are the client's resubscribed set from T002, applied BEFORE the resume query). Replays results in `id` order. **Replay sequence**: server sends `{ type: 'replay_start', count }`, then each event as a `{ type: 'event', event: WSEvent }` message, then `{ type: 'replay_end' }`. **Replay unavailable**: if `lastEventId` is not found in `wsevents` (client was disconnected longer than 5 minutes) OR no event with `id > lastEventId AND expiresat > now()` matches, the engine sends `{ type: 'replay_unavailable', reason: 'event_log_expired' }` and the client falls back to a full REST refresh of affected resources. **Client recovery target**: within 3 seconds of reconnect (under the 5-minute disconnect window). **Cleanup job MUST run** ‚Äî `wsevents` rows that have passed `expiresat` MUST be deleted; they must not accumulate indefinitely. Validate cleanup job throughput under load to confirm 60-second cadence keeps up with high message volume.

### T006 ‚Äî Presence + typing indicators (in-memory, ephemeral)

Implement FR-006. **Presence state** ‚Äî `online` / `offline` / `in_call` ‚Äî is stored in MEMORY only (Map in single-server v1, Redis when running clustered via RedisBroker). NEVER in PostgreSQL. On server restart, presence is rebuilt from reconnecting sessions. **On connect** (T001 successful auth), the engine publishes `presence.online` scoped to all orgs the user belongs to (via spec 011 `orgmemberships` lookup). The presence state is added to the in-memory presence Map keyed by userid. **On disconnect** (or dead-connection cleanup from T004): if the user has NO remaining active sessions, the engine publishes `presence.offline` scoped to all orgs the user belongs to. Multiple-session users stay `online` until the LAST session disconnects. **`presence.in_call`**: published when the user joins a video room (the publisher is spec 001's video service; spec 003 just routes). The `videoroomid` is in the payload. **Initial state fetch**: `GET /api/ws/presence?orgid={id}` returns current presence state for all members of that org ‚Äî used by clients on initial page load BEFORE WebSocket subscriptions are established. Returns `{ orgmemberid ‚Üí status }` map. Visible to authenticated members of the org only (spec 011 `requireAuth({ orgRole: 'member' })`). **Typing indicators** ‚Äî transient, NO persistence: client sends `{ type: 'typing', channelid }` ‚Üí engine publishes `message.typing_start` scoped to `channel:{channelid}`. Server-side timer auto-publishes `message.typing_stop` after **5 seconds** of no repeat typing message from that user. Client can send `{ type: 'typing_stop', channelid }` to cancel early. Typing state is in-memory only ‚Äî no database writes. Typing indicators appear within 100ms of the user starting to type and clear within 5 seconds of them stopping. Acceptable to lose typing state on server restart.

### T007 ‚Äî Capacity monitoring + 70%/90% PA alerts + RedisBroker implementation + GET /api/ws/stats

Implement the FR-007 capacity + alerting portion + the RedisBroker implementation. **`RedisBroker`**: implement the second `WSBroker` from T003 ‚Äî pub/sub via `ioredis`. Subscribes to a shared Redis channel for event publishes; on receipt, routes to local sessions whose subscriptions match the event scope. The `wsevents` log remains in PostgreSQL ‚Äî shared across all server instances in BOTH broker modes. Activated by setting `WS_BROKER=redis` and `REDIS_URL` (no code changes needed ‚Äî already designed-in from T003). **Capacity monitoring**: the engine tracks the current active connection count from `wssessions` every 60 seconds. The capacity ceiling is set via the `WS_MAX_CONNECTIONS` env var (default `1000`). **70% threshold alert**: when active connections ‚â• 70% of `WS_MAX_CONNECTIONS` (default trigger: 700 connections), the engine sends a YappChat PA channel notification via the spec 002 FR-017 `postPANotification` SDK with `bypassQuietHours: true`, `callerscope: 'ws-capacity'`. The message text: `"‚öÝÔ∏è WebSocket capacity warning ‚Äî action recommended. Active connections: {current} of {max} ({pct}% of capacity). The WebSocket engine is approaching its single-process limit. Switch to the Redis broker to support horizontal scaling before connections are refused. To upgrade: set WS_BROKER=redis and REDIS_URL in the server environment and restart. No code changes required. Next check in 24 hours unless connections drop below 70%."` **Recipients**: every user where `users.issystemadmin = true` (spec 011 FR-009). The engine fans the alert to each system admin once per occurrence. If zero system admins exist yet (pre-bootstrap), the alert falls back to a server-side `console.error` log entry with the same content. **The alert is sent at most once per 24 hours** ‚Äî if capacity stays above 70%, the engine does not spam a notification on every check. The alert RE-ARMS (will fire again) only after connections drop below 60% and then rise above 70% again. **90% critical alert** (default trigger: 900 connections) using the same delivery mechanism: `"üö® WebSocket capacity critical ‚Äî immediate action required. Active connections: {current} of {max} ({pct}% of capacity). New connections may be refused soon."` Same 24-hour dedup and re-arm logic. The 70% and 90% alerts are tracked independently. **`GET /api/ws/stats`** (admin only ‚Äî system admin via spec 011): returns `{ activeConnections, maxConnections, pct, broker: 'local' | 'redis', alertThreshold70Triggered, alertThreshold90Triggered, lastAlertSentAt }`. Used by an operator dashboard to monitor capacity.

### T008 ‚Äî Client library WSClient + React integration (WSProvider, useWSEvent, usePresence, useTypingIndicator, WSStatusIndicator)

Build the client-side library and React integration. **`WSClient`** at `src/lib/ws-client.ts` ‚Äî singleton class used by ALL scopes (NOT a React component). Methods: `WSClient.connect(token)` establishes connection and starts the heartbeat loop (T004 application-level ‚Äî send `{ type: 'heartbeat' }` every 25s, treat as dead if no `heartbeat_ack` within 10s); `WSClient.subscribe(scopes[])` sends T002's subscribe message; `WSClient.unsubscribe(scopes[])` sends unsubscribe; `WSClient.on(eventType, handler)` registers an event listener for a specific event type; `WSClient.off(eventType, handler)` removes the listener; `WSClient.sendTyping(channelid)` sends typing with debounce to 2s (so rapid keystrokes don't spam the server ‚Äî actual server-side typing-stop timeout is 5s per T006); `WSClient.disconnect()` performs a graceful close. Property: `WSClient.status` returns `'connecting' | 'connected' | 'reconnecting' | 'disconnected'`. **Reconnection** is automatic with exponential backoff: 1s ‚Üí 2s ‚Üí 4s ‚Üí 8s ‚Üí max 30s. On successful reconnect, the client (a) re-authenticates with a fresh token, (b) re-subscribes to its prior scope set, (c) sends `{ type: 'resume', lastEventId }` (T005) with the last-received event id, (d) dispatches the replay events through normal `on()` handlers so consumers don't need special replay logic. **Token-expiry mid-session safe**: the server doesn't validate the token after connect (T001), so token rotation doesn't drop the session. On the next reconnect, the new token is used. **React integration**: `WSProvider.tsx` at `src/ui/providers/WSProvider.tsx` ‚Äî React context provider that wraps the app, manages the `WSClient` singleton lifecycle (connect on mount, disconnect on unmount), and exposes `useWSClient()` hook. `useWSEvent` at `src/ui/hooks/useWSEvent.ts` ‚Äî hook signature `useWSEvent(eventType, handler, deps[])` subscribes to a specific event type from the nearest `WSProvider`; cleans up listener on unmount. `usePresence` at `src/ui/hooks/usePresence.ts` ‚Äî returns current presence state for a `userid` or `orgid`. Listens to `presence.*` events and keeps a local Map updated; initial state fetched from `GET /api/ws/presence?orgid={id}` (T006). `useTypingIndicator` at `src/ui/hooks/useTypingIndicator.ts` ‚Äî returns the set of userids currently typing in a given `channelid`. Manages typing-start/stop timers locally (auto-clear after 5s of no typing-start to mirror server-side timer). `WSStatusIndicator` at `src/ui/components/ws/WSStatusIndicator.tsx` ‚Äî small icon in the app header showing connection status: green dot (connected), amber spinner (reconnecting), red dot (disconnected). Click shows last disconnect reason (close code + reason text).
