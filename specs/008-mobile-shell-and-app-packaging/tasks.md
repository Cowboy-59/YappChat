### T001 — Expo project scaffolding + custom dev client + EAS build/submit/update pipelines

Implement FR-001 and FR-009. Create `apps/mobile/` workspace package with `expo` as a dependency; the web app remains in `apps/web/` and both consume the same `packages/ui` shared React component library. Configure `app.config.ts` (preferred over `app.json` for typed config): app name `YappChat`, slug `yappchat`, bundle identifier `com.wxperts.yappchat`, Android package `com.wxperts.yappchat`, supported orientations (portrait + landscape on iPad/tablets only — phone-portrait only otherwise), icon and splash assets matching YappChat brand. **ALL native module config goes through Expo plugins** in `app.config.ts` — never edit `ios/` or `android/` directories by hand (bare-workflow native edits are a hard violation). Plugin entries: `@livekit/react-native` (sets `NSCameraUsageDescription`, `NSMicrophoneUsageDescription`, iOS background modes `audio` + `voip`), `expo-secure-store`, `expo-notifications` (declares Apple Push Notifications capability + drops `google-services.json` for Android FCM), `expo-linking`, `sentry-expo`. iOS background modes restricted to `audio`, `voip`, and `remote-notification` — no `fetch` background mode (Apple rejects abuse). `eas.json` defines three EAS Build profiles: `development` (custom dev client, internal distribution), `preview` (release build, internal distribution via TestFlight + Play Internal Testing), `production` (release build, store-bound). Validate the custom dev client builds via `eas build --profile development --platform ios` and `--platform android` and can be installed on a device, then `npx expo start --dev-client` loads the JS bundle into that client. **Release pipelines**: `eas build --profile production --platform ios` produces an `.ipa` ready for App Store Connect; `--platform android` produces an `.aab` ready for Play Console. `eas submit --profile production --platform ios` uploads to App Store Connect for TestFlight + review; `--platform android` uploads to Play Console. `eas update --branch production --message "<msg>"` publishes a JS bundle update — clients on production builds receive it on next launch within 60s of publish. Native changes still require a full build. **EAS secrets are the only acceptable source of API keys, push certs, and signing credentials** (Apple distribution cert, Apple Push Notification key, Google Play service account, Sentry DSN) — committing any of these is a hard violation. CI: each release tag in git triggers a job that bumps `version` in `app.config.ts`, runs `eas build --profile production`, and creates a draft release in App Store Connect / Play Console; final submit is gated on a manual approval.

### T002 — Shared UI reuse: packages/ui + useResponsive + packages/platform adapter

Implement FR-002. Set up `packages/ui` (or equivalent) workspace exporting the components used by both `apps/web` and `apps/mobile`. Set up `packages/platform` as the ONLY place where `react-native-*` packages are imported — web and mobile consume cross-platform abstractions from this package so the web build does not pull in mobile-only code and vice versa. Implement `useResponsive()` hook at `packages/ui/hooks/useResponsive.ts` returning `{ kind: 'mobile' | 'desktop', width, height }` — reads `Platform.OS` on mobile and `window.innerWidth` on web. Drives spec 005 `AIChatPanel` sizing (full-screen on mobile, 33vw on desktop), spec 001 directory layout, etc. **Platform branching is allowed ONLY at leaf component level** via `Platform.OS` checks or `useResponsive()`; the bulk of spec 001–007 component code is unchanged. No spec 001–007 component imports `react-native-*` packages directly — they all go through `packages/platform`. CSS-in-JS or styling that targets DOM-only elements (e.g., `position: fixed`) is replaced with a cross-platform alternative or branched by platform. **The build MUST FAIL if web-only CSS reaches the mobile bundle** — wire a Metro bundler check and/or a CI test that detects DOM-only CSS in mobile bundles. A component that exists in both bundles MUST behave equivalently.

### T003 — MobileLifecycle event stream + WS reconnect + SSE abort + memory_warning

Implement FR-003. Create singleton `MobileLifecycle` module at `apps/mobile/src/lifecycle.ts` that wraps React Native `AppState` and emits typed events: `foreground` (app becomes active), `background` (app moves to background but still resident), `inactive` (transient — call sheet, notification overlay), `memory_warning` (OS asking app to release memory), `appwillterminate` (iOS only — final notice before process kill). Subscribers register with `MobileLifecycle.on(handler)`. Expose `MobileLifecycle.currentState()` so any code can synchronously check foreground vs background — spec 009 push fanout uses this to decide whether to send a push or rely on the live WS. **On every `foreground` event**, the WebSocket client's `resume(lastEventId)` is automatically called by the WS provider — no per-screen wiring needed. Spec 003 `resume(lastEventId)` round-trip MUST complete within 3 seconds on a healthy network. **On every `background` event**, all open SSE streams (PA chat per-session, spec 002 FR-008) are aborted via `AbortController`. Reconnection happens on the next `foreground`. **`inactive` events do NOT trigger reconnection** — they are too transient (e.g., user pulled down notification centre). **`memory_warning` events** trigger the React app to drop cached message history beyond the visible viewport in spec 001 `MessageHistoryView` (FR-014). Foreground/background transitions MUST be observable to the React layer within 100ms.

### T004 — SecureKeyStore: cross-platform secure local storage for E2E private keys

Implement FR-004. Create `SecureKeyStore` at `packages/platform/secure-store.ts` exposing `set(userid, key, value)`, `get(userid, key)`, `delete(userid, key)`, `keys(userid)`, `clearUser(userid)`. **`userid` argument namespaces every entry** so logout of one user never touches another user's keys on the same device. On mobile it delegates to `expo-secure-store` (storage keys prefixed with `<userid>:`); on web it delegates to IndexedDB with one object store per user, values wrapped via `SubtleCrypto`. The interface is identical across platforms. **The X25519 private key generated on first use** (spec 001 FR-011) is written via `SecureKeyStore.set(userid, 'e2e-private-key', base64)` — it is NEVER serialised to disk in plaintext outside the secure store. Storing keys in `AsyncStorage`, `MMKV`, plain IndexedDB without wrapping, `localStorage`, or any non-encrypted local store is a hard violation. **iOS**: the keychain item is configured with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` — the key is not synced to iCloud and is unavailable while the device is locked. **Android**: secure store backed by hardware keystore where available (API 23+); on older devices it falls back to AES-encrypted shared preferences with a key derived from `AndroidKeyStore`. **Logout (spec 011 FR-007)** calls `SecureKeyStore.clearUser(userid)` — removes ALL of that user's namespaced entries in one call, leaving other users on the same device untouched. A second user on the same device starts with a fresh key pair. **Biometric prompt is OPTIONAL in v1** — if config requests `requireBiometric: true`, reading the private key triggers Face ID / Touch ID / fingerprint via `expo-local-authentication`. Default: off. **Lost-key recovery is owned by spec 010** — the shell exposes `KeyRecoveryFlow` (recover from passphrase / recovery code) and `PairNewDeviceQR` (cross-device handoff from an existing trusted device) on launch when no key is found in `SecureKeyStore`.

### T005 — Permission flows with in-context rationale + Settings redirect for denied permissions

Implement FR-005. **No permission may be requested at app launch** — each is requested only when its feature is first used, after an in-app rationale screen. Implement `PermissionRationale` component at `apps/mobile/src/PermissionRationale.tsx` — rendered BEFORE the OS permission sheet; user can choose **Allow** (proceeds to OS sheet) or **Not now** (abandons the action and returns to previous screen). Implement `SettingsRedirectCard` at `apps/mobile/src/SettingsRedirectCard.tsx` — shown when a permission was denied via the OS sheet, with a deep-link card opening the OS Settings app at YappChat's entry: e.g., "YappChat needs camera access to start a video call. Open Settings to enable it." **Permissions used + rationale strings** (all localised; English baseline with i18n hooks): Notifications → triggered by first in-app prompt after auth (behaviour owned by spec 009) → rationale `"Get notified when someone messages you, your PA briefing is ready, or a meeting starts."` Camera → triggered by joining a video call (spec 001 FR-007) → `"Let your team see you on video calls."` Microphone → joining a video call → `"Let your team hear you on video calls."` Photo library / files → attaching a file in AI Chat (spec 005 FR-005) → `"Pick a photo or document to share."` Speech recognition (iOS only) → voice input in AI Chat → `"Talk to your assistant instead of typing."` App-store-required `*UsageDescription` strings on iOS and `<uses-permission>` declarations on Android are configured in `app.config.ts` plugins — never edited by hand. **NOT used in v1**: background location, contacts, calendar (calendar binding goes through OAuth web flow — spec 002 FR-004, not the device's local calendar), microphone-always.

### T006 — Native video stack (LiveKit) + Deep linking router + Force-upgrade gate

Implement FR-006, FR-007, FR-008. **Native video (FR-006)**: install + configure `@livekit/react-native` and `@livekit/react-native-webrtc` via Expo plugins. iOS background modes for `audio` and `voip` are declared so calls survive backgrounding. Joining a room uses the same `Room` / `LocalParticipant` API as the web — code in `packages/ui/video/*` is unchanged. Camera, microphone, and screen share work; screen share on iOS uses `ReplayKit` via the LiveKit broadcast extension; Android uses `MediaProjection`. Picture-in-picture supported on iOS 14+ and Android 8+ — when the app backgrounds during a call, a PiP window shows the active speaker. On dropped network, the room reconnects automatically using LiveKit's built-in reconnect — spec 003 WebSocket reconnect (T003) is independent and happens in parallel. Joining a video call from cold start ≤ 5 seconds on 4G. **CallKit (iOS) and ConnectionService (Android) integration is OUT OF SCOPE for v1** — calls do not appear in the OS recents list; acknowledged gap. **Deep linking (FR-007)**: register custom URL scheme `yappchat://` in `app.config.ts` (`scheme: 'yappchat'`). Configure universal links on iOS and App Links on Android for `https://yappchat.app/*` (domain configurable per deployment — NOT hard-coded). The associated `apple-app-site-association` and `assetlinks.json` files are served from the web origin. Implement `DeepLinkRouter` at `apps/mobile/src/DeepLinkRouter.tsx` listening to `Linking.addEventListener('url', ...)` and on cold-start `Linking.getInitialURL()`. **URL surface**: `yappchat://` (or `https://yappchat.app/`) → home; `yappchat://conversation/:conversationid?messageid=:id` → specific conversation optionally scrolled to message; `yappchat://session/:sessionid?messageid=:id` → specific AI Chat session (spec 002 FR-008); `yappchat://pa/notification/:notificationid` → PA channel scrolled to notification; `yappchat://video/:videoroomid` → join video room; `yappchat://settings/avatar` → avatar picker (spec 007); `yappchat://settings/providers` → AI provider manager (spec 002 FR-002). Parser dispatches typed events: `{ kind: 'conversation', conversationid, messageid? }`, etc. Unknown URLs route to home with a non-blocking toast `"We couldn't find that page"`. **Cold-start deep links** — the React app subscribes to the router BEFORE the first render so cold-start launches navigate immediately rather than flashing the home screen. An E2E test covers cold-start race regression on every release. Deep links open the correct surface 100% of the time. **Force-upgrade gate (FR-008)**: on launch, after splash, BEFORE the React app mounts, the shell calls `GET /api/mobile/version` with headers `X-Client-Platform: ios|android` and `X-Client-Version: <semver>`. Response: `{ minSupported, latest, forceUpgrade: boolean, upgradeMessage }`. If `forceUpgrade: true`, render `ForceUpgradeModal` — full-screen blocking modal with title, message, and a single button opening `https://apps.apple.com/app/idXXXXXXXX` (iOS) or `market://details?id=com.wxperts.yappchat` (Android); user CANNOT dismiss. If below `latest` but above `minSupported`, show `UpgradeBanner` (non-blocking) once per session with **Update** and **Later** actions (Later dismisses for 24 hours). **If `/api/mobile/version` is unreachable, the shell allows the app to launch normally** — outage of this single endpoint MUST never lock users out.

### T007 — Crash reporting (Sentry) + session telemetry with PII redaction

Implement FR-010. Configure `sentry-expo` with a DSN per environment (dev / preview / production) stored in EAS secrets. Native crashes (Objective-C / Swift / Kotlin) and JS exceptions are both captured. **Sourcemaps are uploaded to Sentry as part of every EAS build** so stack traces in production are unminified. **User identifiers attached to events are pseudonymous** — `userid` only, NEVER name or email. PII redaction is configured at the Sentry SDK level. **Message content, attachment filenames, and AI prompts are NEVER attached to events** — implement client-side scanning of event payloads that drops any field matching a known content path before submission to Sentry (allowlist over denylist where possible). Session telemetry — app launch, foreground/background transition counts, deep-link source — is sent to a YappChat-controlled endpoint `POST /api/mobile/telemetry`. **Volume is bounded**: at most one batch per minute per device. Implement `Telemetry` module at `apps/mobile/src/telemetry.ts` — throttled batch sender that drops events when offline and flushes on next foreground. **Telemetry is opt-out via a setting**; default is on. The opt-out preference is stored locally (NOT in `SecureKeyStore` — it must work before auth completes) so it persists across launches even pre-auth. Google Play data safety form MUST accurately disclose every category of data collected — telemetry and `mobiledevices` (T008) must be reflected; a privacy review before first submission is required.

### T008 — Per-device install registry + AuthGate mount + linking points for spec 009 / spec 011

Implement FR-011 and FR-012. **`mobiledevices` table** per project DB conventions (lowercase, no separators, UUID v7 PK, FK names match parent + `id`): columns `userid`, `deviceid` (text — client-generated UUID stored in `SecureKeyStore` — same value as spec 001 `userencryptionkeys.deviceid`), `platform` enum `ios`/`android`, `model` (e.g. `iPhone15,2`, `Pixel 8 Pro`), `osversion`, `appversion` (semver), `locale` (BCP 47 e.g. `en-US`), `timezone` (IANA e.g. `America/Denver`), `lastseenat`, `createdat`. **UNIQUE constraint on `(userid, deviceid)`**. Generate migration via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. Endpoints: `POST /api/mobile/devices` — on first authenticated launch, body `{ deviceid, platform, model, osversion, appversion, locale, timezone }`; server returns the row with a server-assigned `id`. `PATCH /api/mobile/devices/:id` — heartbeat on every subsequent launch updating `lastseenat`, `appversion`, `osversion`; throttled to once per 24 hours. `DELETE /api/mobile/devices/:id` — on logout; spec 011 FR-007 logout invokes this AND `SecureKeyStore.clearUser(userid)` AND `DELETE /api/push/tokens` (spec 009) in the same atomic sequence so the device is fully decoupled from the signed-out user. `GET /api/mobile/devices` — caller's registered devices, surfaced in a settings screen so users can review where YappChat is installed. `GET /api/admin/mobile/devices?userid=:id` — admin only (system admin per spec 011), returns all devices for a user, used by support to confirm an install before troubleshooting. **Cross-spec `deviceid` contract**: the value generated client-side and stored in `SecureKeyStore` matches `mobiledevices.deviceid` ↔ `userencryptionkeys.deviceid` (spec 001) ↔ `pushtokens.deviceid` (spec 009) ↔ `keypairings.deviceid` (spec 010) ↔ `devicesessions.deviceid` (spec 011 FR-014) — one device, one E2E key pair, one push registration, one session. **Linking points (FR-012)**: `expo-notifications` is installed + configured (Apple Push Notifications capability on iOS, `google-services.json` for FCM on Android) but the shell registers NO handlers in v1 — spec 009 wires the handlers in. `MobileLifecycle.currentState()` (T003) is callable from anywhere — spec 009 uses it to decide whether a server-side WS event needs a push fanout or the user is already foreground. The auth flow boundary is the React component `AuthGate` (implementation owned by spec 011 FR-011 at `packages/ui/src/auth/AuthGate.tsx`). **The shell mounts `MobileRoot` at `apps/mobile/src/MobileRoot.tsx`** — native `RootView` mount point that subscribes to `MobileLifecycle`, mounts `DeepLinkRouter`, calls the force-upgrade gate, then renders `AuthGate` → app. Dev environments use a seeded `users` row (spec 011 bootstrap) instead of any stub-userid helper. Cold start to first interactive screen ≤ 4 seconds on iPhone 12 / Pixel 6 or newer.
