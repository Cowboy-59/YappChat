### T001 — PA core: paconfigs + agent registration + notification bubble triggers + OpenAIChatPanel action contract

Implement FR-001 + the PA setup lifecycle. Create `paconfigs` table per project DB conventions (lowercase, no separators, UUID v7 PK): `userid` UNIQUE, `agentid` FK → spec 001 `agents.id` (the PA's agent registration), `activeproviderid` FK → aiproviders.id (T002), `briefingtimeutc` time (daily briefing time in UTC), `monitorintervalmin` integer (default 5), `notificationprefs` jsonb (thresholds per notification type), `bubbletimeoutms` integer (default 8000 — `PANotificationBubble` auto-dismiss), `createdat`, `updatedat`. **`name` and `avatarurl` are intentionally ABSENT** — spec 007 `avatarconfigs` is the source of truth for both. The PA always renders with the user's resolved avatar persona. Generate migration via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. **Endpoints**: `POST /api/pa/setup` registers the PA on first setup — calls `POST /api/engine/agents` (spec 001 FR-010), passing `name` and `avatarurl` resolved by reading spec 007 `GET /api/avatar/current` for the calling user. Creates the internal PA channel. `GET /api/pa/config` returns current config. `PATCH /api/pa/config` updates briefing time, notification prefs, bubble timeout — does NOT accept `name` or `avatarurl` (those live in spec 007 `PATCH /api/avatar/user`). `DELETE /api/pa/setup` deregisters PA, disconnects all bindings, archives PA channel, soft-deletes all `assistantsessions` (T005) for the user. **Interaction entry points** — two ways to open the PA, both dispatch the SAME client-side action `OpenAIChatPanel`: (1) notification bubble click → `{ source: 'bubble', notificationId }`; (2) PA avatar click → `{ source: 'avatar' }`. Spec 005 subscribes to this action (NOT a direct component coupling — spec 002 owns only the trigger; spec 005 owns the panel surface; spec 007 owns the avatar rendering). **PA avatar status badge** — `idle` / `monitoring` / `responding` / `briefing_delivered` / `error` — broadcast via spec 003 `pa.status` WSEvents scoped `user:{userid}`. **Frontend components**: `PAAvatar.tsx` (wraps spec 007 `<AvatarDisplay size={64}>` with status-badge overlay + unread notification dot; click → dispatches `OpenAIChatPanel({ source: 'avatar' })`), `PANotificationBubble.tsx` (compact floating card in bottom-right — spec 007 avatar thumbnail `size={24}` + first line of notification text + dismiss × button; click NOT × → dispatches `OpenAIChatPanel({ source: 'bubble', notificationId })`; auto-dismisses after `paconfigs.bubbletimeoutms`), `PANotificationBubbleStack.tsx` (container managing the queue — max 3 simultaneous, oldest at bottom; each new one animates in from bottom-right; `"+ N more"` overflow indicator when more than 3 are pending; dismissed bubbles do NOT mark notification as read — only removes the visual bubble), `PAMessageBubble.tsx` (single message row used inside spec 005's panel AND inside the PA channel feed — renders text + structured cards + download links + tool-call indicators).

### T002 — AI provider registry + adapter layer (OpenAI-compatible / Anthropic / Ollama / Custom) + admin-only system default

Implement FR-002. Create `aiproviders` table: `userid` nullable (providers are per-user; NULL when `isdefault: true` for system-default), `name` (user label e.g. `Local Llama3`, `Work Claude`), `type` enum `openai-compatible` / `anthropic` / `ollama` / `custom`, `baseurl`, `model` (e.g. `llama3`, `claude-sonnet-4-6`, `gpt-4o`), `apikeyref` (reference to stored secret — NEVER the key itself in plaintext), `supportstooluse` boolean, `supportsstreaming` boolean, `isdefault` boolean, `lastpingedat`, `lastpinglatencyms`, `createdat`. **Single-default enforcement**: partial unique index `CREATE UNIQUE INDEX aiproviders_one_default ON aiproviders ((1)) WHERE isdefault = true` — at most one row may have `isdefault: true`. Generate migration; validate. **Adapter layer**: define `PAProviderAdapter` interface — `chat(messages, tools?): AsyncIterable<Delta>` + `selectTool(messages, tools): Promise<ToolCall | null>` (fallback for non-tool providers). Implementations: **`OpenAICompatibleAdapter`** (single adapter covers OpenAI, Ollama, LM Studio, Groq, vLLM, any `/v1/chat/completions` endpoint), **`AnthropicAdapter`** (native `@anthropic-ai/sdk`, maps tool-use to the interface), **`CustomAdapter`** (user provides a request/response transformer function). When `supportstooluse: false`, the PA falls back to a two-step pattern: first generates a tool selection decision as plain text, then parses and invokes — identical user behaviour, slower response. **Endpoints**: `GET /api/pa/providers` lists registered providers with connectivity status. `POST /api/pa/providers` body `{ name, type, baseurl, model, apikeyref, supportstooluse, supportsstreaming }` — on registration, engine pings the provider's chat endpoint with a test message; returns `{ connected: true, latencyms }` or an error. **The per-user route MUST REJECT any `isdefault` field in the body** — only `PATCH /api/pa/providers/system-default` can write that flag. `PATCH /api/pa/providers/:id` updates provider config. `DELETE /api/pa/providers/:id` removes — cannot delete the active provider without switching first. `POST /api/pa/providers/:id/ping` tests connectivity. `PATCH /api/pa/providers/system-default` body `{ providerid }` or `{ providerid: null }` — **GATED to callers with `users.issystemadmin = true` (spec 011 FR-009)**; clearing the flag from previous default, setting on the new one, all in one transaction so the partial unique index never sees two rows with `isdefault: true`. `PATCH /api/pa/config` with `{ providerid }` switches the active provider — takes effect on the next message, NO restart, under 5 seconds to switch. **Frontend**: `PAProviderManager.tsx` (list / add / edit / test / delete AI providers; set active provider; the `isdefault` toggle only visible to system admins via spec 011 `hasSystemFlag('issystemadmin')`).

### T003 — Proactive monitoring loop + panotifications + dashboard + postPANotification internal SDK + pasystemnotificationaudit

Implement FR-003, FR-009, FR-017. Create `panotifications` table: `userid`, `type` enum `briefing` / `pending_messages` / `calendar_reminder` / `project_overdue` / `email_alert` / `avatar_conversion_complete` / `keybackup_recovery_alert` / `ws_capacity_warning` / `mcp_server_offline` / `system`, `payload` jsonb (structured notification data), `read` boolean default false, `createdat`. Create `pasystemnotificationaudit` (30-day retention via `expiresat`): `id`, `userid`, `type`, `callerscope` (e.g. `ws-capacity`, `keybackup-recovery`, `pa-monitoring`), `createdat`, `expiresat`. **Monitoring loop** runs on `paconfigs.monitorintervalmin` (default 5 min). Each tick checks: unread message counts per channel (from spec 001 `messages` + `conversations.lastreadmessageid`), mentions of the user, upcoming calendar events within 15 minutes, project items newly overdue (wxKanban MCP `projecttasks` where `duedate < now()`). Each notification type has a user-configurable threshold from `paconfigs.notificationprefs`. The loop MUST be supervised — automatic restart on crash; dead-man alert to system admins after 2 missed cycles. PA avatar status badge updates to `monitoring` during the polling loop and `idle` between polls (spec 003 `pa.status`). **Notifications posted as PA messages** in the PA channel with `messagetype: 'notification'` and structured payload `{ type, count, items[], timestamp }`. Every posted notification ALSO triggers a `PANotificationBubble` (T001) — **exception**: if spec 005's `AIChatPanel` is already open AND the user's scroll position is at the bottom (actively reading), the bubble is suppressed. Notifications deduplicated — the same unread count is NOT posted twice within a single monitoring window. A notification is marked `read: true` automatically when the user opens `AIChatPanel` and scrolls past it OR clicks the bubble that referenced it. Morning briefing fires a bubble: `"Good morning — your briefing is ready."` **Dashboard (FR-009)**: `GET /api/pa/dashboard` returns `{ pendingMessages: { total, byChannel: [{ channelid, name, count, mentions }] }, projectItems: { overdue, dueToday, dueThisWeek }, nextEvent: { title, startsIn } }`. PA renders this as a `PADashboardCard` in its channel on morning briefing and on demand. **Endpoints**: `GET /api/pa/notifications` returns last 100 PA-generated notifications with read/unread; `PATCH /api/pa/notifications/:id/read` marks one read; `POST /api/pa/notifications/read-all` marks all read. **`postPANotification` internal SDK (FR-017)** at `packages/server/pa-notify/index.ts` — exports `postPANotification(opts: { userid, type, payload, previewtext, bypassQuietHours? }): Promise<{ notificationid }>`. Writes a `panotifications` row, then publishes a `pa.notification` WSEvent scoped `user:{userid}` via spec 003's `WSBroker.publish`. **This SDK is the ONLY supported way for non-PA server code to add to the PA channel** — direct INSERT into `panotifications` from another scope is a hard violation. Per-user rate limit: 60 system-posted notifications per hour; excess logs a warning and returns without writing. Each call writes a `pasystemnotificationaudit` row to attribute floods to the responsible `callerscope`. `bypassQuietHours: true` sets a flag on the WSEvent payload that spec 009 FR-007 reads when deciding whether to push during quiet hours. The push fanout (spec 009) picks up the WSEvent automatically — no extra wiring needed. Same SDK surface in both `LocalBroker` and `RedisBroker` modes. **Frontend**: `PADashboardCard.tsx`, `PANotificationFeed.tsx` (scrollable list of PA-generated notifications with read/unread state), `PAPendingSummary.tsx` (compact widget in main sidebar — total pending messages + next meeting + overdue item count), `PANotificationSettings.tsx` (per-type thresholds + briefing time).

### T004 — Calendar + Email OAuth bindings + shared OAuth callback handler + token refresh job

Implement FR-004, FR-005, FR-016. Create `pacalendarbindings` table: `userid`, `provider` enum `google` / `outlook` / `caldav`, `accountemail`, `tokenref` (encrypted OAuth token reference — NEVER plaintext), `scopes` text[], `createdat`. Create `paemailbindings`: `userid`, `provider` enum `gmail` / `outlook` / `imap`, `accountemail`, `tokenref`, `vipsenders` text[] (always trigger notifications), `keywords` text[] (subject/body keywords that trigger notifications), `createdat`. **OAuth library decision gate** before implementation: evaluate Better Auth (MIT, 30+ built-in providers, recommended for ≤ 30 providers), Arctic (MIT, 50+ providers, zero-dependency, edge-runtime-compatible), or Nango (Elastic License 2.0, 700+ providers, self-hosted Docker — choose if provider count > 30 or white-label connect UI is a priority). The chosen library determines how `tokenref` storage and refresh are managed — must be consistent across all providers. **Shared OAuth callback (FR-016)**: `GET /api/integrations/oauth/callback/:provider` accepts standard OAuth 2.0 query params (`code`, `state`, `error?`). `:provider` matches a key in the OAuth provider registry. The `state` parameter is a signed token created when `POST /api/pa/calendar/bind` or `/email/bind` starts the flow — encodes `userid`, `bindingtype`, and a 10-minute expiry. Mismatched or expired state → HTTP 400, no token storage. On success, exchanges code for tokens, stores `accesstoken` and `refreshtoken` as references in the secrets store, writes the binding row, redirects user back to a deep link the PA channel sends them to. All redirect URIs registered with each provider in advance — callback MUST validate `state.expectedredirect` matches the post-callback redirect target. **Calendar (FR-004)**: `POST /api/pa/calendar/bind` initiates OAuth binding (Google Calendar, Microsoft Outlook, CalDAV). `GET /api/pa/calendar/bindings` lists connected accounts. `DELETE /api/pa/calendar/bind/:id` disconnects. `GET /api/pa/schedule?view=day|week|month&date=<ISO>` returns structured schedule data: events with title, start, end, location, attendees; project items due in the range interleaved by date. Natural language schedule queries via PA chat (handled in T005): `"What do I have on Thursday?"`, `"Schedule a 30-minute call with the engineering team on Friday afternoon"`. **Event creation, update, deletion via PA chat — PA ALWAYS confirms the action before committing** (constraint: PA MUST NOT send email or create calendar events without explicit user confirmation). **Email (FR-005)**: `POST /api/pa/email/bind` initiates OAuth binding (Gmail, Outlook, IMAP). PA monitoring loop checks for new emails from VIP senders OR matching user-defined keywords; posts a notification when found. Natural language email queries via PA chat. **PA can draft email replies and show them to the user for approval before sending** — NEVER sends without explicit user confirmation per message. **Token refresh job**: background job runs every 10 minutes scanning bindings whose access token expires within 15 minutes; refreshes via the provider's refresh endpoint; updates the secret. On refresh failure (revoked / expired refresh token), binding marked `status: 'revoked'` and PA posts via `postPANotification`: `"Your Google Calendar connection needs to be reconnected"`. **Frontend**: `PACalendarSettings.tsx` (connect/disconnect accounts, view binding status), `PAEmailSettings.tsx` (connect/disconnect email, configure VIP senders + keywords).

### T005 — Named multi-turn AI Chat sessions backend + SSE streaming + context window mgmt + PA-driven content creation via spec 006

Implement FR-008 + FR-007. Create `assistantsessions` table: `userid`, `name` (default timestamp like `"May 10, 4:32pm"`), `providerid` FK → aiproviders.id (frozen at create — user can re-create a session against a different provider), `lastmessageat` (updated on every message append — drives sidebar ordering), `deletedat` nullable (soft-delete; hard-deleted by daily job 30 days after this is set), `createdat`. Index on `(userid, deletedat NULLS FIRST, lastmessageat DESC)` for the session list query. Create `assistantmessages` table: `sessionid` FK → assistantsessions.id, `role` enum `user` / `assistant` / `tool_result`, `encryptiontype` enum `e2e` / `platform` (mirrors spec 001), `content` text NULL when `encryptiontype = 'e2e'`, `encryptedpayload` bytea set when `e2e`, `encryptionkeyid` FK → spec 001 `userencryptionkeys.id` for E2E rows, `attachmentids` uuid[] (FK array → spec 005 `chatattachments.id`, nullable), `toolcalls` jsonb nullable (for `role = 'assistant'` — structured record of skills invoked during this turn: `{ skillid, arguments digest, success, latencyms }`), `subagentexecutionid` uuid nullable (set when this assistant turn spawned a subagent), `prompttokens` integer nullable, `completiontokens` integer nullable, `createdat`. Index on `(sessionid, createdat DESC)` for cursor pagination. Generate migrations; validate. **`assistantmessages` follow the SAME E2E rules as spec 001 `messages`** — server stores ciphertext only for `encryptiontype: 'e2e'`. The 20-turn context window sent to the AI provider is assembled CLIENT-SIDE (browser decrypts then forwards plaintext to the configured provider) — server NEVER sees plaintext for E2E sessions. **Endpoints**: `POST /api/pa/sessions` body `{ name? }` creates new session. `GET /api/pa/sessions` returns caller's sessions ordered by `lastmessageat DESC`, soft-deleted excluded — each row includes session id, name, last message preview (first 60 chars), `lastmessageat`, `createdat`. `GET /api/pa/sessions/:id` returns session detail + current provider label (so spec 005 renders `"Powered by Llama3 (local)"`). `PATCH /api/pa/sessions/:id` renames. `DELETE /api/pa/sessions/:id` soft-deletes. `GET /api/pa/sessions/:id/messages` cursor-paginated history — `before` ISO timestamp, `limit` max 100 default 50. `POST /api/pa/sessions/:id/messages` body `{ content, attachmentids?: string[] }` — response is `text/event-stream` with delta events: `token`, `tool_call_start`, `tool_call_end`, `subagent_spawned`, `context_truncated`, `done`. **Per-session context window**: each request to the AI provider includes the last 20 messages of `assistantmessages` for that session. If the provider's context limit is exceeded, drop the oldest messages first AND emit a `context_truncated` SSE event so the client can show a notice. **Streaming token delivery** uses spec 003 WebSocket `pa.status` events for the avatar state machine (spec 007 listens); the SSE channel itself is the message-content delivery path. **Session export (FR-008)**: `GET /api/pa/sessions/:id/export?format=markdown|pdf|txt` returns the rendered session — implements spec 005 FR-007. PDF delegates to spec 006 via `POST /api/gen/pdf` with the session content. **PA channel proactive routes**: `POST /api/pa/messages` (used by the PA backend itself to post into the PA channel — NOT for user-initiated chat), `GET /api/pa/messages` PA channel message history (briefings, alerts) cursor-based. **FR-007 content creation via PA**: when the user asks for a presentation / document inside a session, the PA assembles content via registered skills (e.g., `fetch_channel_messages`, `get_project_status`), then calls spec 006's API (`POST /api/gen/presentation`, `POST /api/gen/pdf`, etc.) with the assembled content + format. PA presents an OUTLINE for user approval BEFORE generating the full document — user can modify structure through conversation. On approval, PA triggers generation, posts a progress note (`"Generating your presentation, ~30 seconds"`), then posts the download link via `GeneratedFileCard` (spec 006 T008) when complete. PA can iterate: `"Add a slide on incidents"` → re-runs generation with updated outline. Presentation delivered as a downloadable file within 60 seconds of user approval.

### T006 — Skill invocation runtime (FR-014) + subagent execution runtime (FR-015) + tool-use → handler HTTP call + status WSEvents

Implement FR-014 and FR-015. Spec 004 explicitly excludes the runtime; it lives here. Create `skillinvocations` table: `skillid` FK → spec 004 `skills.id`, `userid`, `sessionid` nullable FK → assistantsessions.id, `subagentexecutionid` nullable FK → subagentexecutions.id, `invokedby` enum `pa` / `subagent` / `studio_test`, `arguments` jsonb (tool-use arguments sent to the handler), `httpstatus` integer nullable when network error, `responsebody` jsonb nullable truncated to 32KB, `errormessage` nullable, `latencyms`, `success` boolean (true for 2xx), `invokedat`. Index on `(skillid, invokedat DESC)` — drives spec 004 stats + the `> 20% error rate over last 7 days` warning. Create `subagentexecutions` table: `agenttemplateid` FK → spec 004 `agenttemplates.id`, `userid`, `parentsessionid` nullable FK → assistantsessions.id, `prompt`, `status` enum `spawned` / `working` / `waiting_for_input` / `completed` / `error`, `result` jsonb nullable on completed, `errormessage` nullable, `prompttokens`/`completiontokens` nullable, `runtimems` nullable, `startedat` (set when status moves to `working`), `completedat` nullable, `createdat`. Generate migrations; validate. **Skill invocation runtime (FR-014)**: when the AI provider returns a tool-use block selecting a skill `name`, the PA resolves the matching `enabled: true` row from spec 004's `skills` table. **Disabled skills produce a `skill_disabled` SSE error event and are NOT invoked**. Validate tool-use arguments against `skills.inputschema` (JSON Schema Draft 7, `ajv`). Validation failures return a `tool_result` with `{ ok: false, error: 'input_validation_failed', details }` to the AI provider — the model can correct and retry. **Synchronous skills** (`async: false`): POST `{ arguments }` to `handlerurl` with headers `X-Skill-Token: <skilltoken>`, `Content-Type: application/json`, `User-Agent: yappchat-pa/<version>`. Timeout: **30 seconds**. Response body returned to the AI provider as the `tool_result`. **Asynchronous skills** (`async: true`): spawn a subagent via FR-015 and return a `tool_result` containing `{ subagentid, status: 'spawned' }` immediately — model continues without waiting. Every invocation writes one `skillinvocations` row. **Retry policy**: on HTTP 5xx or network error, retry up to 2 times with 1s/3s backoff. On HTTP 4xx, do NOT retry — return the error to the model immediately. **Rate limiting**: per-skill concurrency cap of 5 in-flight invocations per user. Excess queue for up to 10 seconds, then fail with `tool_result.error: 'skill_busy'`. **Skill handler URLs MUST be HTTPS in production** — runtime rejects HTTP URLs with `error: 'insecure_handler_url'` unless `ALLOW_HTTP_SKILL_HANDLERS=true` (matches spec 004 constraint). **Subagent execution runtime (FR-015)**: `POST /api/pa/subagents` body `{ agenttemplateid, prompt, parentsessionid? }` spawns a subagent — template's `enabled` flag must be true (disabled → HTTP 422). Subagent runs as a background task (Node.js worker / queued job) with its own AI provider context built from template's `systemprompt` + spawn prompt. **Skill set available to the subagent is EXACTLY the template's `agenttemplateskills` join — NOT the user's full catalog**. Status changes pushed via spec 003 WebSocket `subagent.status` events scoped `user:{userid}` — spec 005 renders this in `SubagentCard`. Skill invocations made by a subagent write `skillinvocations` rows with `invokedby: 'subagent'` AND `subagentexecutionid` set. On completion, final response written to `subagentexecutions.result` (jsonb) + `subagent.status: 'completed'` event emitted. **If `parentsessionid` is set, an `assistantmessages` row is appended to that session with the result** so the user sees it inline. On error, `errormessage` set + `subagent.status: 'error'` emitted. **Hard runtime limit of 10 minutes** (configurable per-template via `agenttemplates.maxruntimeseconds` from spec 004, range 60-3600). Exceeding the limit moves the row to `status: 'error'` with `errormessage: 'runtime_limit_exceeded'`. Endpoints: `GET /api/pa/subagents/:id` current state. `GET /api/pa/subagents` list — filter `?sessionid=` or `?status=`. `POST /api/pa/subagents/:id/cancel` — sets status to `error` with `errormessage: 'cancelled'`.

### T007 — Community Skills publish/browse/install/update + setup-guide library with step-by-step guidance

Implement FR-006 (community skills) + FR-010 (setup guidance). Create `communityskills` table: `authorid`, `sourceskillid` FK → spec 004 `skills.id` (author's original private skill), `name` UNIQUE snake_case, `label`, `description`, `category` enum, `inputschema` jsonb, `handlerurl`, `async` boolean, `version` semver, `downloadcount` integer, `published` boolean (false when author unpublishes — existing installations unaffected), `createdat`, `updatedat`. Create `communityskillversions` (immutable diff history): `communityskillid` FK, `version`, `previousversion`, `changesummary` text (author-written or PA-generated), `schemadiff` jsonb `{ added, removed, modified }`, `handlerurlchanged` boolean, `publishedat`. Create `communityskillsubscriptions`: `userid`, `communityskillid` FK, `localskillid` FK → skills.id (user's installed local copy), `installedversion`, `skippedversion` nullable (most recent update declined), `updatedat`, `createdat` — UNIQUE on `(userid, communityskillid)`. **Endpoints**: `GET /api/pa/skills` returns user's private skill catalog with `communityskillid` set if installed from community. `POST /api/pa/skills/register` (PA-conversational creation flow — delegates to spec 004's `POST /api/studio/skills`); tagged `createdby: 'pa'`. **Interactive skill creation via PA conversation**: user describes skill → PA asks clarifying questions (endpoint, inputs, output, sync/async, category) → PA generates full skill definition → user approves → PA calls register; skill `visibility: 'private'` default. PA can suggest browsing Community Skills BEFORE creating a duplicate. **Publish to Community Skills**: `POST /api/pa/skills/:id/publish` body `{ category }` — copies skill to `communityskills` with user as `authorid`, `version: '1.0.0'`, discoverable to all. `GET /api/community/skills` returns public catalog filterable by `category` sortable by `downloadcount` or `createdat` — results within 500ms. `GET /api/community/skills/:id` detail. `GET /api/community/skills/:id/versions` full history. `POST /api/community/skills/:id/install` copies skill definition into user's `skills` table + creates `communityskillsubscriptions` row — installed copy immediately usable by PA. `PATCH /api/community/skills/:id` (author publishes update) increments `version`, writes `communityskillversions` diff record, fires `postPANotification` (T003) to every subscriber with the update prompt: `"The community skill {name} has been updated v{prev} → v{new} by @{author}. What changed: {changesummary}. Would you like to update your copy? Reply yes / no / diff."` (spec 003 `skill.update_available` event also published). User replies `yes` → `POST /api/community/skills/:id/update-installed` overwrites local copy, sets `updatedat`. User replies `no` → sets `skippedversion`; user NOT re-prompted for that version. User replies `diff` → PA posts full before/after JSON schema comparison + re-asks yes/no. `DELETE /api/community/skills/:id` (author unpublishes) clears subscriptions; PA notifies each subscriber that their installed copy is now independent. **Setup-guide library (FR-010)**: create `setupguides` table: `guidekey` UNIQUE slug (e.g. `google-calendar`, `anthropic-api-key`, `ollama-local-install`, `mcp-github`), `providername`, `connectiontype` enum `oauth` / `apikey` / `url-credentials` / `local-install`, `version` semver, `steps` jsonb (ordered array of `{ stepnumber, title, instruction, url?, screenshotdesc? }`), `notes` nullable, `updatedat`. Create `pasetupprogress` (per-user resume state): `userid`, `guidekey` FK, `laststepnumber`, `completed` boolean, `startedat`, `updatedat` — UNIQUE `(userid, guidekey)`. Seed guides for: Google Calendar, Microsoft Outlook Calendar, CalDAV, Gmail, Microsoft Outlook Email, IMAP, Anthropic, OpenAI / compatible, Ollama local install, Custom AI provider, Skill handler endpoint. Endpoints: `GET /api/pa/setup-guides` list all. `GET /api/pa/setup-guides/:key` full guide. `GET /api/pa/setup-guides/:key/progress` user's current progress. `PATCH /api/pa/setup-guides/:key/progress` `{ laststepnumber }`. `POST /api/pa/setup-guides/:key/reset`. **Guidance behaviour**: every binding flow surfaces a `"How do I set this up?"` button — clicking it OR a natural-language equivalent in the PA channel (`"How do I get a Google Calendar API key?"`, `"What is a client secret?"`) triggers `SetupGuideCard` rendering. **Steps broken into the smallest meaningful actions** — each fits one line. PA monitors user progress through the conversation — if user says `"I'm on the credentials page"`, PA picks up from that point. **Sensitive values (client secrets, API keys) are NEVER typed into the PA channel** — PA instructs the user to paste them into the secure input field in `OAuthSetupGuide` / `SecureInputPrompt` instead. PA channel only receives a confirmation. Resume from last completed step on return. **Frontend**: `CommunitySkillBrowser.tsx`, `CommunitySkillCard.tsx`, `CommunitySkillDetail.tsx`, `SkillPublishFlow.tsx`, `SkillUpdatePromptCard.tsx` (Yes/No/Show Diff), `SkillDiffView.tsx`, `CommunitySkillVersionHistory.tsx`, `SetupGuideCard.tsx`, `SetupStep.tsx`, `SecureInputPrompt.tsx`, `SetupResumeCard.tsx`, `OAuthConsentRedirect.tsx`.

### T008 — MCP server integration: registration + tool aggregation + persistent connections + offline reconnect

Implement FR-011. Create `mcpservers` table: `userid`, `name` (user-assigned label), `transport` enum `stdio` / `http` / `sse`, `command` nullable (for stdio — full shell command e.g. `npx -y @modelcontextprotocol/server-filesystem /home/docs`), `url` nullable (for http/sse base URL), `headersref` nullable (reference to stored secret containing auth headers JSON — NEVER plaintext), `enabled` boolean, `status` enum `connected` / `offline` / `error`, `lastconnectedat`, `lasterrorat` nullable, `lasterrormessage` nullable, `createdat`. Create `mcptooldiscovery` (cached tool/resource/prompt definitions): `mcpserverid` FK, `itemtype` enum `tool` / `resource` / `prompt`, `name`, `description`, `inputschema` jsonb nullable (null for resources/prompts), `uri` nullable (resource URI), `mimetype` nullable, `discoveredat` — UNIQUE on `(mcpserverid, itemtype, name)`. Generate migrations; validate. **Use `@modelcontextprotocol/sdk` already present in the YappChat `mcp-server` workspace package** — reuse from the workspace rather than installing a separate copy. **On registration** (`POST /api/pa/mcp` body `{ name, transport, command?, url?, headers?, enabled }`): PA backend connects using `@modelcontextprotocol/sdk` `Client` class and calls `list_tools()`, `list_resources()`, `list_prompts()`. Results cached in `mcptooldiscovery`. **At PA startup**, the engine connects to all `enabled: true` MCP servers and maintains persistent connections — `stdio` servers kept as subprocesses, `http`/`sse` servers maintain transport session. **Tool aggregation**: all tools from all connected MCP servers are merged with the registered `skills` table when building the AI provider tool list. **De-duplicated by tool name — MCP tools take precedence over skills with the same name, with a warning logged**. **MCP tool names MUST be unique across all sources** (registered skills + all MCP servers) in the merged tool list. The PA MUST surface name conflicts in `MCPServerManager` so admins can rename the conflicting skill. **MCP tool invocation**: when the AI provider selects an MCP tool, PA calls the server's `call_tool(name, arguments)` method and returns the result as a `tool_result` — IDENTICAL flow to skill invocation (T006). **Endpoints**: `GET /api/pa/mcp` lists registered servers with status + tool count + last connected. `POST /api/pa/mcp` registers new server. `GET /api/pa/mcp/:id` server detail. `PATCH /api/pa/mcp/:id` updates config. `DELETE /api/pa/mcp/:id` removes — disconnects subprocess or HTTP session, deletes discovery cache. `POST /api/pa/mcp/:id/reconnect` triggers immediate reconnect. `GET /api/pa/mcp/:id/tools` cached tool list. `GET /api/pa/mcp/:id/resources` cached resource list with URI / name / description / MIME type. PA can fetch a resource's content via `read_resource(uri)` and include in context. `POST /api/pa/mcp/:id/discover` re-runs tool/resource/prompt discovery — refreshes cache. **Disconnect handling**: if an MCP server disconnects, PA marks `status: 'offline'`, removes its tools from the merged tool list, posts via `postPANotification` (T003): `"MCP server {name} went offline. Its tools are temporarily unavailable."` PA reconnects automatically with exponential backoff. **Security**: MCP server secrets (API keys in headers) stored as secret references in `mcpservers.headersref` — NEVER in plaintext. `stdio` MCP servers run as subprocesses with the SAME OS user as the PA service — deployment must ensure the PA process has no more permissions than needed. **Setup guidance**: FR-010 applies to MCP servers — if user asks `"How do I connect the GitHub MCP server?"`, the PA provides step-by-step instructions using setupguides keys `mcp-github`, `mcp-filesystem`, etc. **Well-known MCP servers** the PA can connect to out of the box (any MCP-compatible server works): `@modelcontextprotocol/server-filesystem`, `@modelcontextprotocol/server-github`, `@modelcontextprotocol/server-slack`, `@modelcontextprotocol/server-postgres`, `@modelcontextprotocol/server-brave-search`, or any custom MCP server. **Frontend**: `MCPServerManager.tsx` (settings panel — list, Add Server button, connect/disconnect/delete per server), `MCPServerCard.tsx` (single server row — name + transport badge + connection status dot + tool count + last connected + expandable tool list), `MCPServerStatusBadge.tsx`, `MCPToolList.tsx` (expandable per server), `MCPAddServerForm.tsx` (transport selector + conditional fields + test connection button), `MCPOfflineNotificationCard.tsx` (rendered in PA channel when a server goes offline — server name + error + Reconnect Now button).
