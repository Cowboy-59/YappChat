## Task Details

### T001 — Core data model + opaque-token primitives + requireAuth middleware

Create all 13 tables this spec owns per project DB conventions (lowercase, plural, no separators, UUID v7 PKs, FK names match parent + `id`): `users` (with three system-flag booleans, `kind`, `plan` enum, `passwordhash` nullable), `sessions` (opaque hashed token, sliding 24h expiry, deviceid nullable), `refreshtokens` (hashed token, `familyid`, `replacedbyid` self-FK, 30-day TTL), `magiclinktokens`, `passwordresettokens`, `emailverificationtokens`, `orgmemberships` (UNIQUE(userid, orgid), role enum), `orginvitations`, `devicesessions` (UNIQUE(userid, deviceid, sessionid)), `agentapitokens` (with `last6chars` for display), `oauthproviderconfigs` (column MUST be `clientsecretref`, NEVER `clientsecret`; CI test rejects), `oauthidentities` (UNIQUE(providerkey, subjectid)), `authauditlog` (append-only, 90-day TTL, anonymised IPs). Generate migrations via `npm run db:generate` and validate with `wxkanban-agent dbpush --dry-run`. Implement argon2id hashing helpers with m_cost 64MB / t_cost 3 / parallelism 1, configurable via `AUTH_ARGON2_MEMORY_KIB` / `AUTH_ARGON2_ITERATIONS` env. Implement SHA-256 token hashing (sessions, refresh, magic, reset, verification, agent, invitation). Implement session-token issuance (32 random bytes from `crypto.randomBytes`, base64-URL, httpOnly + Secure + SameSite=Lax + Path=/ cookies) and validation in ≤ 5ms p95 via one indexed lookup on `sessions(sessiontokenhash)`. Implement the single `requireAuth({ orgRole?, systemFlag? })` middleware — bespoke per-route auth checks are a hard violation. Implement org-scope filter middleware that injects `orgid` from `orgmemberships` into every org-scoped query (skills, sessions, aiproviders, panotifications, pushtokens, oauthidentities, agentapitokens, messages). Implement audit-log writer with anonymised IPs (last octet zeroed for v4; last 80 bits for v6). NO JWTs anywhere — opaque tokens only.

### T002 — Email+password signup/login + email verification + password reset

Implement FR-001, FR-002, FR-006. `POST /api/auth/signup` accepts `{ email, password, displayname, plan, orgname? }` — `plan: 'individual' | 'corporate'` required; `orgname` required only when `plan === 'corporate'`; missing `plan` → 400 `{ error: 'plan_required' }`, missing `orgname` on corporate → 400 `{ error: 'orgname_required_for_corporate' }`. Server-side floor: password length ≥ 8 (client also enforces zxcvbn ≥ 3). On success: create `users` row with `plan`, `orgs` row (name from `orgname` for corporate OR `"{displayname}'s Workspace"` for individual), set `orgs.seatlimit = 1` + `plantype='individual'` for individual plans OR seatlimit null + `plantype='corporate'` for corporate, owner `orgmemberships` row, initial session, refreshtoken (familyid=id), emailverificationtoken (24h TTL). Email uniqueness via citext UNIQUE — duplicates return 422 with generic body (no enumeration via timing or message). Sends verification email linking to `GET /api/auth/email-verify/:token`. Verification is required for high-trust actions but NOT for sign-in. Soft rate limit: 10 signups per IP per hour → 429. Signup → first authenticated screen ≤ 10s p95. `POST /api/auth/login` accepts `{ email, password }` — verifies argon2id in constant time, returns 200 + cookies + `{ user, org }`. Rate limit: 5 wrong attempts per (IP + email) per 15 min → 429 with `retry-after`. Wrong email and wrong password MUST return the SAME error code and SAME response timing. Writes `authauditlog` row `eventtype: 'login'`. `POST /api/auth/password-reset/request` always returns 202 (no enumeration); creates `passwordresettokens` row (15-min TTL, single-use, hashed). `POST /api/auth/password-reset/consume` re-hashes new password, updates `users.passwordhash`, AND revokes ALL existing sessions + refresh-token families for that user. `POST /api/auth/email-verify/request` resends; `GET /api/auth/email-verify/:token` consumes; round-trip ≤ 60s including real email send.

### T003 — Magic-link / email-OTP passwordless login

Implement FR-003. `POST /api/auth/login/magic/request` body `{ email }` — ALWAYS returns 202 regardless of whether the email exists (no account enumeration). If the email matches a user, create a `magiclinktokens` row (10-minute TTL, single-use, SHA-256 `tokenhash` stored; plaintext is the URL segment in the emailed link). The token lookup MUST be by SHA-256 hash; plaintext NEVER leaves the email body. `GET /api/auth/login/magic/:token` validates, marks `consumedat`, issues session + refresh tokens, redirects to the app. Replayed tokens return 410; expired tokens return 410. Frictionless onboarding: if no `users` row matches the email, a new account is created on consume (with auto-personal-org + owner membership, identical to signup), so first-time visitors can sign up by magic link without ever picking a password. Email arrives within 30s p95; click-to-app under 3s. Writes `authauditlog` row `eventtype: 'login'` on consume.

### T004 — Refresh-token rotation with reuse detection + logout + WS propagation + SecureKeyStore.clearUser

Implement FR-005 and FR-007. `POST /api/auth/refresh` validates the presented refresh token by hash, generates a NEW refresh token, issues a new session, sets `refreshtokens.replacedbyid = <newid>` on the old row. Implement a 5-second grace window: the immediately-previous token is also accepted within 5s of rotation (returns the same new tokens) to handle network retry races without false-positive family revoke. **Reuse detection**: if a request presents a refresh token whose `replacedbyid IS NOT NULL` AND the grace window has expired, revoke the ENTIRE `familyid` (every refreshtokens row + every linked sessions row) in a single DB statement ≤ 1s, write `authauditlog` with `eventtype: 'family_revoke'`, publish `auth.signed_out` WSEvent scoped `user:{userid}` via spec 003, AND call spec 002 FR-017 `postPANotification` with `bypassQuietHours: true`, `callerscope: 'auth-family-revoke'`, previewtext "We logged you out everywhere because we detected a reused login token. Sign back in." Single-token rotation without family revoke is a hard violation. `POST /api/auth/logout` revokes the calling session (`revokedat`), revokes the linked refresh-token family, publishes `auth.signed_out` scoped `user:{userid}`, writes audit row, clears cookies (`Set-Cookie: session=; Max-Age=0`). Server-side revoke + WSEvent emit ≤ 500ms p95. Client-side on successful logout response: call `SecureKeyStore.clearUser(userid)` (spec 008 FR-004) to purge all of that user's E2E keys from the device. Other tabs/devices receive the WSEvent and trigger their own client-side logout within 2s. PA notification arrives within 5s of family revoke.

### T005 — Org membership + invitations + system flags + bootstrap admin

Implement FR-008, FR-009, FR-010. `orgmemberships.role` enum (`owner` / `admin` / `member`) with UNIQUE(userid, orgid). Owners can promote/demote/transfer/delete-org; admins can invite + change member↔admin but NOT touch other admins or owners; members are read-only on management. **Sole-owner protection**: demoting OR deleting the last `owner` of an org returns HTTP 422 `{ error: 'last_owner_protection' }`. `POST /api/orgs` creates org + owner membership for caller. `POST /api/orgs/:orgid/invitations` (owner/admin) creates `orginvitations` (7-day TTL, hashed token, pre-set role), emails accept link. `POST /api/orgs/invitations/:token/accept` consumes; if email is not yet a users row the recipient must sign up first (invitation is bound to email, not userid). `PATCH /api/orgs/:orgid/memberships/:userid` changes role respecting hierarchy. `DELETE /api/orgs/:orgid/memberships/:userid` cascades clean-up of org-scoped resources owned by that user IN that org. `GET /api/orgs/:orgid/memberships` visible to any member. Three system flags (`issystemadmin`, `isbillingadmin`, `issupport`) — independently grantable booleans defaulting to false. `PATCH /api/auth/system-roles/:userid` restricted to `issystemadmin`; writes audit `eventtype: 'role_grant'`. `GET /api/auth/system-roles` lists users with any flag (visible to `issystemadmin` or `issupport`). **Bootstrap**: on server startup, if `BOOTSTRAP_ADMIN_EMAIL` env is set AND `SELECT count(*) FROM users WHERE issystemadmin=true` is 0, INSERT a new users row OR UPDATE the matching-email row to set both `issystemadmin=true` AND `isbillingadmin=true`. Audit row with `eventtype: 'role_grant'`, `payload: { reason: 'bootstrap' }`. Bootstrap is IDEMPOTENT — re-runs with a system admin already present are a no-op. On bootstrap, granted user receives BOTH a PA notification (spec 002 FR-017) AND an email (cold-start coverage when PA channel isn't yet configured).

### T006 — Force sign-out + device session registry + agent API tokens + spec 010 pairing prerequisite

Implement FR-013, FR-014, FR-015, FR-016. `devicesessions` table joins userid ↔ deviceid ↔ sessionid; the deviceid value MUST match across spec 008 `mobiledevices.deviceid`, spec 009 `pushtokens.deviceid`, spec 001 `userencryptionkeys.deviceid`, and spec 010 `keypairings.deviceid` — multi-tenant fuzz test verifies a single deviceid joins across all 4 spec scopes. `GET /api/auth/sessions` returns the caller's active sessions (`id`, `deviceid`, device summary, anonymised IP, `createdat`, `lastusedat`). `POST /api/auth/sessions/:id/revoke` lets the caller revoke their own session. `POST /api/auth/sessions/:id/force-revoke` is admin force sign-out: authorization is org-admin caller AND target session's user is in caller's org, OR system-admin caller (anywhere). On force-revoke, revoke the session, mark `devicesessions.revokedat`, emit `auth.force_signout` WSEvent scoped `user:{targetuserid}` with `{ sessionid, deviceid, by: callerid, reason? }`. Propagation to target device ≤ 2s p95 via spec 003. Target device's AuthGate drops to LoginScreen + runs `SecureKeyStore.clearUser(userid)`; OTHER devices for the same user are unaffected. Audit row `eventtype: 'force_signout'` with full payload including caller. **Agent API tokens** (FR-015): `POST /api/auth/agents/:agentid/tokens` issues a token; plaintext returned ONCE in response, `tokenhash` + `last6chars` stored. Authorization: caller is registrar OR org admin OR system admin. `agents` joins `users` via `userid` where `users.kind='agent'`; agent auth via `Authorization: Bearer <apitoken>` middleware resolves to that users row. `DELETE /api/auth/agents/:agentid/tokens/:tokenid` sets `revokedat`; NEXT request rejected — no caching delay. `GET /api/auth/agents/:agentid/tokens` lists tokens showing `last6chars` only. **Spec 010 pairing prerequisite** (FR-016): `POST /api/keybackup/pairing/start` returns 401 if unauthenticated. The `pairing:{pairingid}` WS scope subscription checks `keypairings.userid = currentuserid` AND a valid session. The pairing QR nonce is signed with the server's auth secret so the existing device verifies the QR was issued by this server (prevents pairing-QR spoofing).

### T007 — OAuth/OIDC providers + login + link/unlink (Google, Apple, Microsoft, GitHub, generic OIDC)

Implement FR-017, FR-018 using the spec 002 FR-016 OAuth library decision (Better Auth / Arctic / Nango — same library powers both `/api/integrations/oauth/...` and `/api/auth/oauth/...`). Seed `oauthproviderconfigs` on first launch: `google`, `apple`, `microsoft`, `github` with `enabled=false`, `clientid` + `clientsecretref` blank. `clientsecretref` is a reference into the secrets store — plaintext storage is a hard violation; CI test rejects schemas naming the column `clientsecret`. `GET /api/auth/oauth/providers` returns enabled providers (no secrets). `GET /api/auth/oauth/:provider/start` runs auth-code + PKCE: generate `state` (32 random bytes, httpOnly cookie), generate `code_verifier` + `code_challenge`, redirect to provider. `GET /api/auth/oauth/callback/:provider` validates state, exchanges code + verifier, fetches profile (email, sub, name, picture). On first OAuth login (no matching `oauthidentities` row): create users row with `passwordhash=NULL`, `emailverifiedat=now()` IFF provider asserts `email_verified=true` ELSE NULL, plus personal org + owner membership. Resolve subsequent logins via `oauthidentities(providerkey, subjectid)`. **SOC 2 NO AUTO-LINK BY EMAIL**: if provider's email matches an existing email-password account, return HTTP 409 "An account already exists for this email. Sign in with your password first, then link this provider from settings." Linking is an explicit authenticated user action only — prevents account takeover via attacker-controlled OAuth account. End-to-end OAuth login ≤ 5s p95 (excluding provider's UI). Generic OIDC: `PATCH /api/auth/oauth/providers/:providerkey` with `{ issuerurl, clientid, clientsecretref }` — discover endpoints via `/.well-known/openid-configuration`. **Link/unlink** (FR-018): `POST /api/auth/oauth/:provider/link` (authenticated) initiates link flow; callback handler reads `state.intent='link'`, adds the identity to the calling user instead of creating an account. Already-linked-to-different-user → 409. `GET /api/auth/oauth/identities` returns caller's linked identities. `DELETE /api/auth/oauth/identities/:id` unlinks BUT 422 if the unlink would leave the user with no sign-in method (no password AND no other linked identity). Linking/unlinking write `eventtype: 'oauth_link'` / `'oauth_unlink'` audit rows.

### T008 — Frontend auth surface: AuthGate, useAuth/AuthContext, login/signup/reset forms, account-management UIs, WS listeners

Implement FR-011, FR-012, plus every component under `packages/ui/src/auth/`. `AuthGate` (`packages/ui/src/auth/AuthGate.tsx`) is the root above every client surface: fetches `GET /api/auth/me` on mount, shows brand-aware skeleton during the initial fetch (avoids flash of unauthenticated content), renders `LoginScreen` if unauthenticated OR children + `AuthContext` if authenticated, subscribes to `auth.signed_out` and `auth.force_signout` WSEvents scoped `user:{currentuserid}`. Cold render ≤ 200ms p95 (single `/me` round-trip). Spec 008 mobile shell mounts AuthGate as root per spec 008 FR-012. `AuthProvider` + `useAuth` (`packages/ui/src/auth/AuthContext.tsx`) — React Context with shape `{ user, org, hasRole(name: 'owner'|'admin'|'member'): boolean, hasSystemFlag(flag: 'issystemadmin'|'isbillingadmin'|'issupport'): boolean, signIn(method, payload), signOut(), refresh() }`. `hasRole` checks active-org role; `hasSystemFlag` checks users.* boolean. Reactive — all consumers re-render on auth state change. No per-component auth fetches. Build: `LoginScreen` (tabbed: email+password / magic link; `OAuthButtonRow` below; forgot-password link), `SignupForm` (email + password with zxcvbn meter + display name + plan/orgname fields per spec 012 FR-006 routing), `LoginForm` (rate-limit UX showing `retry-after` countdown), `MagicLinkForm`, `PasswordResetRequestForm`, `PasswordResetForm`, `EmailVerificationBanner` (persistent when `emailverifiedat IS NULL` with Resend action), `OAuthButtonRow` (one button per enabled provider, rendered in both LoginScreen and SignupForm). Account/session: `DeviceSessionsList` (caller's sessions with Revoke per row), `LinkedIdentitiesPanel` (linked OAuth identities + sign-in-method count, Link/Unlink — Unlink button disabled when it would be the last method), `OrgInviteAcceptScreen` (linked from invitation email; signed-out users sign up first), `OrgMembershipManager` (owner/admin invite + role change + remove, with sole-owner protection UX), `SystemRoleManager` (`issystemadmin`-only, toggles the three system flags). Listeners mounted inside AuthGate: `ForceSignoutListener` (triggers logout on matching device for `auth.force_signout`), `RefreshFamilyRevokedListener` (instant logout on `auth.signed_out`). Self-signout footgun mitigation: the force-revoke UI confirms with the user's email + "you are signing yourself out from this device" warning if the target session is the caller's own.
