## Task Details

### T001 — Skill CRUD + skills table + enable/disable lifecycle + auto-versioning + skillversions table + rollback

Implement FR-001 and FR-005. Create tables per project DB conventions (lowercase, no separators, UUID v7 PKs, FK names match parent + `id`): **`skills`** (`name` unique within org snake_case 1-64 chars, `label`, `description` sent to AI providers, `category` enum `productivity`/`communication`/`data`/`development`/`finance`/`media`/`integration`/`custom`, `inputschema` jsonb JSON Schema Draft 7, `handlerurl`, `skilltoken` STORED HASHED (bcrypt) — never returned in API responses after creation, `async` boolean, `enabled` boolean default false, `version` semver text, `createdby` enum `studio`/`pa`/`import`, `communityskillid` nullable FK, `createdat`, `updatedat`); **`skillversions`** (immutable diff history — `skillid` FK, `version`, `previousversion`, `changedfields` text[], `schemadiff` jsonb `{ before, after }`, `updatedat`, `updatedby`). Generate migrations via `npm run db:generate`; validate with `wxkanban-agent dbpush --dry-run`. **Endpoints**: `POST /api/studio/skills` creates a skill — starts `enabled: false` REGARDLESS of creation method. Returns the plaintext `skilltoken` ONCE in the response; hashes it before storage. `GET /api/studio/skills` returns full org catalog filterable by `category`/`enabled`/`async`/`createdby` and searchable by `name`/`label`/`description` — results within 500ms. `GET /api/studio/skills/:id` returns full skill detail including usage stats summary (T006) and version history summary. `PATCH /api/studio/skills/:id` updates editable fields. **Each save changing `inputschema` or `handlerurl` auto-increments semver patch version AND writes a `skillversions` record**. Users may request minor or major bump via `PATCH` with `{ versionbump: 'minor' | 'major' }`. `DELETE /api/studio/skills/:id` permanently deletes — BLOCKED if invoked within the last 24 hours (check `skillinvocations.invokedat`); requires explicit `{ override: true }` confirmation flag. `PATCH /api/studio/skills/:id/enable` and `disable` toggle `enabled` immediately — NO restart. Disabled skills are instantly removed from the PA tool list. **Name uniqueness enforced at API layer** (NOT only DB constraint — for clean error messages). Skill handler URLs MUST be HTTPS in production; HTTP only permitted when `ALLOW_HTTP_SKILL_HANDLERS=true` (dev only) — enforced server-side. `GET /api/studio/skills/:id/versions` returns full history newest-first. `POST /api/studio/skills/:id/rollback` body `{ version }` restores `inputschema` and `handlerurl` from that version as a NEW version record — does NOT delete history. **Frontend (SkillStudio shell)**: `SkillStudio.tsx` (resizable split sidebar + detail panel), `SkillList.tsx` (scrollable catalog showing name, category badge, enabled toggle, error-rate warning badge from T006, last used), `SkillForm.tsx` (full editor — embeds `JSONSchemaEditor` from T002; save auto-versions), `SkillVersionHistory.tsx` (timeline with version + date + changed fields + expandable diff + Rollback button per entry), `SkillMetricsBadges.tsx` (compact inline badges — success rate %, avg latency, last used; YELLOW warning when error rate > 20% — visible without querying the DB per row).

### T002 — JSON Schema editor + skill handler test console + skilltestlogs

Implement FR-002 and FR-003. Create `skilltestlogs` table: `skillid` FK → skills.id, `testedinput` jsonb (arguments sent), `httpstatus` integer, `responsebody` jsonb (full response), `latencyms`, `success` boolean (true if 2xx), `testedat`, `testedby` userid. Generate migration; validate. **JSON Schema editor (FR-002)**: build `JSONSchemaEditor.tsx` — visual JSON Schema Draft 7 editor. Supports: field add/remove, name, type (`string` | `number` | `boolean` | `array` | `object`), required toggle, description, enum values, number `min`/`max`, string `minLength`/`maxLength`. Editor shows a LIVE preview of the generated JSON Schema as the user builds it (side pane with formatted JSON). Schemas validated with `ajv` before saving — invalid schemas rejected with plain-English error (NOT raw ajv output). **Test console (FR-003)**: `POST /api/studio/skills/:id/test` sends a POST to the handler URL with test arguments and `X-Skill-Token` header. Returns `{ status, latencyms, responseBody, requestSent }` REGARDLESS of success or failure — the caller always sees the full exchange. Test timeout: 30 seconds (same as live invocations). Test results stored in `skilltestlogs`. Test invocations write to spec 002's `skillinvocations` table tagged `invokedby: 'studio_test'` — distinct from live PA calls (`invokedby: 'pa'`); shared runtime path so coverage is identical. **Test console UI**: `SkillTestConsole.tsx` — auto-generated input form from the skill's `inputschema` (each input field rendered with the appropriate widget based on type), Send button, response panel showing exact JSON sent + HTTP status code + response body (syntax-highlighted with `highlight.js`) + round-trip latency. **Input validation BEFORE handler call**: validate test inputs against the current `inputschema` and show per-field errors inline — NOT a raw 422 from the server. Unreachable handler shows clear error: `"Handler URL not reachable — check the URL and that your server is running"`. `GET /api/studio/skills/:id/testlogs` returns the test history visible in the skill detail. The handler authenticates using `X-Skill-Token` (generated on creation, shown once, stored hashed per T001).

### T003 — Handler code generation (TypeScript / Python / JavaScript) + Deploy checklist

Implement FR-004. `POST /api/studio/skills/:id/generate-handler` body `{ language: 'typescript' | 'python' | 'javascript' }` returns the handler source as `{ source: string, filename: string, language }`. **TypeScript handler**: Express route, `X-Skill-Token` validation, request body typed from `inputschema` (generate the TypeScript interface from the JSON Schema), response typed as `{ result: unknown }`, try/catch error handling with structured error response. **Python handler**: FastAPI route, `X-Skill-Token` header check, Pydantic model auto-generated from `inputschema` (use `datamodel-code-generator`-style derivation or hand-roll). **JavaScript handler**: plain Express route (no types), `X-Skill-Token` check, jsdoc comments for input shape derived from `inputschema`. **Every generated handler MUST include the auth pattern** verbatim: ```const token = req.headers['x-skill-token']; if (!token || token !== process.env.SKILL_TOKEN) return res.status(401).json({ error: 'Unauthorized' });``` (equivalent in Python via FastAPI dependency). Generated code includes **inline comments explaining each section** — written for a developer unfamiliar with YappChat skill handlers. Comments cover: token validation, input parsing, where to put business logic, response shape, error handling. **`HandlerCodeGenerator.tsx` UI**: language selector (radio buttons or tabs for TS / Py / JS), generated code displayed with syntax highlighting (via `highlight.js`), copy-to-clipboard button, **Deploy checklist** panel showing the steps as a numbered list with checkboxes (1) Deploy handler to a public HTTPS URL, (2) Verify it is reachable from the YappChat server (`curl -X POST <url> -H 'X-Skill-Token: <token>' -d '<test-input>'`), (3) Paste URL back into skill form, (4) Run a test from `SkillTestConsole`. Generation is read-only — it does NOT deploy. The developer is responsible for deploying the generated handler.

### T004 — Agent template CRUD + agenttemplates + agenttemplateskills + maxruntimeseconds + skill-validity guard

Implement FR-006. Create tables: **`agenttemplates`** (`name`, `description`, `avatarurl`, `systemprompt` (full system prompt text), `providerid` FK → spec 002 `aiproviders.id`, `async` boolean, `enabled` boolean, **`maxruntimeseconds`** integer DEFAULT 600 with range 60-3600 — hard runtime cap that spec 002 FR-015 enforces, `createdat`, `updatedat`). **`agenttemplateskills`** join table: `agenttemplateid` FK → agenttemplates.id, `skillid` FK → skills.id. UNIQUE constraint on `(agenttemplateid, skillid)`. Generate migrations; validate. **Endpoints**: `POST /api/studio/agents` creates a template with `name`, `description`, `avatarurl`, `systemprompt`, `providerid`, `async`, `maxruntimeseconds` (validate 60-3600 range — out-of-range returns HTTP 422), `skillids[]`. `GET /api/studio/agents` lists all templates showing skill count, async flag, enabled state. `GET /api/studio/agents/:id` returns template detail including resolved skills and stats (T006). `PATCH /api/studio/agents/:id` updates any field; skill list changes take effect on next spawn. `DELETE /api/studio/agents/:id` BLOCKED if an active `subagentexecutions` row references this template (read-only check against spec 002's table). `PATCH /api/studio/agents/:id/enable` and `disable` toggle. **Skill-validity guard** (critical): templates may include ONLY `enabled: true` skills — blocked at the API layer on create AND on enable. If a user attempts to enable a template whose skill set contains a now-disabled skill, return HTTP 422 with `{ error: 'template_contains_disabled_skills', disabledSkillIds }`. **The known v1 gap is acknowledged**: if a skill assigned to an enabled template is disabled AFTER the template is enabled, spawned agents will fail to invoke that skill — flagged as a follow-on scope item, not blocked. **Frontend (AgentStudio shell)**: `AgentStudio.tsx` (split-view template list sidebar + detail panel), `AgentTemplateList.tsx` (avatar, name, async badge, skill count, enabled toggle), `AgentTemplateForm.tsx` (name, avatar upload via spec 007 picker integration, description, system prompt textarea, AI provider picker reading from spec 002 `aiproviders`, async toggle, `maxruntimeseconds` number input with 60-3600 range validation client-side, skill multi-select from the T001 catalog filtered to `enabled: true` only).

### T005 — Agent template test console + sandbox runner + agenttestlogs

Implement FR-007. Create `agenttestlogs` table (7-day retention via `expiresat`): `agenttemplateid` FK → agenttemplates.id, `testprompt`, `toolcalls` jsonb (ordered array of `{ skillname, input, output, latencyms }`), `responsetext`, `prompttokens`, `completiontokens`, `totallatencyms`, `testedat`, `testedby` userid. Generate migration; validate. **`POST /api/studio/agents/:id/test`** body `{ prompt }` creates a TEMPORARY sandbox PA session, runs the template against the test prompt, and returns: response text, full tool-call trace (skill name → inputs → output per call), token counts (prompt / completion), total latency. **Sandbox tests stored in `agenttestlogs`** — NOT in spec 002's `assistantsessions` (sandbox runs don't pollute the user's real session history). 7-day retention via a daily cleanup job that purges rows past `expiresat`. **Unavailable AI provider**: if the template's `providerid` references a provider that is unreachable or has invalid credentials, return a clear error `{ error: 'provider_unavailable', providerName }` rather than hanging or returning a generic 500. Skill calls during the sandbox run go through spec 002's runtime (FR-014) tagged `invokedby: 'agent_test'` so the runtime path is shared but the source is distinguishable in `skillinvocations`. The sandbox respects `agenttemplates.maxruntimeseconds` — if the test run exceeds the cap, abort and return the partial trace with `status: 'timeout'`. `GET /api/studio/agents/:id/testlogs` returns sandbox test history (last 7 days). **`AgentTestConsole.tsx` UI**: test prompt textarea, Run button, results panel showing: response text (with markdown rendering), collapsible tool-call trace per skill call (each call expandable to show inputs / outputs / latency), token counts (prompt / completion / total cost estimate), total latency. While the test is running, show a live spinner with the current step (`"Thinking…"`, `"Calling skill {name}…"`, `"Generating response…"`) updated by WebSocket events from spec 003.

### T006 — Skill import/export bundles + validation preview + skill/agent metrics aggregation

Implement FR-008 and FR-009. **Skill import/export (FR-008)**: `GET /api/studio/skills/export` returns all org skills as a portable JSON bundle: `{ version: 1, exportedAt, skills: [...] }`. **`skilltoken` values are NEVER exported** — the bundle includes a placeholder requiring the importer to mint a new token on import. `POST /api/studio/skills/import` body `{ bundle }` validates each skill definition against the JSON Schema for the bundle format AND validates each skill's `inputschema` is a valid JSON Schema Draft 7. Returns a preview `{ toCreate: SkillSummary[], toSkip: { name, reason }[], errors: { name, error }[] }` — NO creation happens at this step. `POST /api/studio/skills/import/confirm` body `{ bundleid }` (returned from the preview call) creates all `toCreate` skills in `enabled: false` state, tagged `createdby: 'import'`. New `skilltoken` minted per skill and returned ONCE in the response. **Metrics (FR-009)** — read-only from spec 002's tables (this spec NEVER writes to `skillinvocations` or `subagentexecutions`): `GET /api/studio/skills/:id/stats` returns `{ totalInvocations, successRate, avgLatencyMs, p95LatencyMs, errorCount, lastInvokedAt }` aggregated from spec 002 `skillinvocations` rows where `skillid` matches. `GET /api/studio/agents/:id/stats` returns `{ totalSpawns, completedCount, errorCount, avgRuntimeMs, lastSpawnedAt }` aggregated from spec 002 `subagentexecutions` rows. **Error-rate warning**: skills with error rate > 20% over the LAST 7 DAYS are highlighted with a warning badge in `SkillList` (T001) — implement as a materialised aggregate or a fast query path (cached for 1 minute) so the catalog can be rendered without per-row stats queries. **`SkillImportFlow.tsx` UI**: file picker for the bundle JSON → validation results table showing `Create` / `Skip` / `Error` per skill with reason → Confirm button (disabled until at least one `toCreate` is selected). On confirm, show the list of newly minted tokens once — the user must save them before closing the dialog (re-display is not possible since they're hashed at rest).

### T007 — Studio Assistant (Archie) backend: persona config + similarity search + guided chat + studioassistantconfig

Implement FR-010 backend. The Studio Assistant is a conversational AI assistant — the **Studio Assistant** (default persona name `Archie`) — embedded as a persistent avatar in the Studio sidebar. Before any creation begins, Archie runs a similarity search against the community skills/agents catalog and presents the result with three clear paths. Creation then proceeds conversationally: Archie fills the form live as the user answers questions. **Persona config**: store in `studioassistantconfig` table — `id`, `personaname` default `"Archie"`, `avatarurl` (configurable image — defaults to a robot/wrench icon), `tone`, `role` text. `GET /api/studio/assistant/config` returns the current persona config. `PATCH /api/studio/assistant/config` (admin only via spec 011 — `hasSystemFlag('issystemadmin')` OR org admin) updates persona name, avatar, provider. **AI provider routing**: Archie is powered by the same AI provider the user has configured for their PA (spec 002 `paconfigs.activeproviderid`). If the user has no PA set up yet, Archie falls back to the **system default provider** — the first `aiproviders` row flagged with `isdefault: true`. Deployments configure exactly one default provider at install time; if none is configured, `POST /api/studio/assistant/chat` returns HTTP 503 `{ error: 'no_assistant_provider_configured' }` and `StudioAssistant` UI (T008) shows: `"Set up a personal AI provider to use Archie."` **Similarity search (FR-010)**: `POST /api/studio/assistant/describe` body `{ description, type: 'skill' | 'agent' }` runs semantic similarity against the community catalog using embedding comparison (or AI-powered description matching against `communityskills.description` + `communityskills.name`). Returns `{ matches: [{ communityskillid, name, label, similarity, installcount }], topMatch?: {...} }`. **Results returned within 3 seconds**. If `topMatch.similarity >= 0.75` (configurable threshold, env `STUDIO_SIMILARITY_THRESHOLD`), the UI presents the three-path card. If no match above threshold, the UI immediately starts the guided creation flow. **Guided creation chat**: `POST /api/studio/assistant/chat` body `{ history, formstate, message, type }` (where `type` is `skill` or `agent`) returns `{ reply, fieldUpdates: { fieldname: value, ... }, suggestedNextQuestion, readyToSave? }`. Archie asks a FIXED sequence of clarifying questions, each mapped to a skill/agent field. As the user answers, Archie writes the value into `fieldUpdates` — the UI applies these to the live form in real time. **Skill creation question sequence** (verbatim phrasing): (1) `"What should this skill be called? Give me a short snake_case name — like 'send_slack_message' or 'get_weather'."` (2) `"Describe what it does in 1–2 sentences, as if explaining to the AI assistant that will use it."` (3) `"What category fits best? [shows category list]"` (4) `"What inputs does it need? List them — I'll handle the JSON Schema. Example: 'city name (required text), units (optional, metric or imperial)'."` (5) `"What URL should I call to run it? (Leave blank for now if you haven't deployed yet)"` (6) `"Should this run instantly (synchronous) or in the background as a subagent (asynchronous)?"` (7) `"Ready to save? I'll keep it disabled until you've tested the handler."` Agent template questions follow the same pattern. **Every Archie response includes `suggestedNextQuestion`** so the UI can show a one-click reply button. **`readyToSave`** is set when Archie has collected enough information to produce a complete, valid skill/agent definition. **Archie NEVER saves, enables, or publishes anything without explicit user confirmation** — the confirmation step is the `SkillConfirmationCard` in T008. **Conversation history is session-local** — NOT stored in the database after the Studio tab is closed (no `studioassistantsessions` table). The Studio MUST accept an optional `initialDescription` prop so the spec 005 AI Chat (FR-008) can pass the user's message directly to Archie without requiring re-entry — Archie immediately fires the similarity search with this description and pre-fills the conversation.

### T008 — Studio Assistant frontend: StudioAssistant sidebar + SimilarityMatchCard + LiveFormHighlight + SkillConfirmationCard

Implement FR-010 frontend. Build all components under `src/ui/components/studio/`. **`StudioAssistant.tsx`** — persistent sidebar panel present in BOTH `SkillStudio` and `AgentStudio`. Shows Archie's avatar (configurable from `studioassistantconfig` — T007), name, status dot (`thinking` / `ready`), and the conversation thread. **Always visible** — does NOT require navigating away from the form. The panel is fixed-width on the right edge of the Studio (collapsible on mobile). Accepts an optional `initialDescription` prop from spec 005 AI Chat — when provided, automatically fires the similarity search on mount as if the user had submitted that description. **`StudioAssistantBubble.tsx`** — single message bubble in the assistant thread. Archie messages LEFT-aligned with avatar; user replies RIGHT-aligned. Renders plain text, `SimilarityMatchCard` when a community match is found, and `SkillConfirmationCard` when Archie is ready to save. **`SimilarityMatchCard.tsx`** — rendered by Archie when a community match is found (similarity ≥ 0.75). Shows: matched skill name, author handle (e.g., `@sasha`), install count (e.g., `"147 installs"`), one-line description, similarity confidence badge (visualised as a percentage or a confidence bar). Three action buttons: **Use as-is** (installs via spec 002 `POST /api/community/skills/:id/install` — no new skill created), **Start from this** (copies community skill into a new private draft with all fields pre-filled, `communityskillid` set, `enabled: false`; user then customises with Archie's guidance), **Build from scratch** (ignores the match, Archie starts the guided creation flow from step 1). **`StudioAssistantInput.tsx`** — message composer at the bottom of the assistant panel. Text field, Send button, and a `SuggestedReply` chip showing Archie's `suggestedNextQuestion` — clicking the chip populates AND sends in one tap. **`LiveFormHighlight.tsx`** — wrapper applied to any form field that was just updated by Archie via the `fieldUpdates` payload from `/api/studio/assistant/chat`. Briefly highlights the field with a pulse animation (200ms ease-out, fade over 1s) so the user can see what changed. The Studio applies `fieldUpdates` to the live form in real time AS each Archie response arrives — the user can SEE the form fill in as they chat. The user can switch to the form at any time to edit directly; manual edits sync back to Archie's `formstate` on next message. **`SkillConfirmationCard.tsx`** — rendered by Archie (via `StudioAssistantBubble`) when `readyToSave: true` in the chat response. Shows a summary of ALL skill/agent fields in a read-only card. Two buttons: **Save (keep disabled)** (calls `POST /api/studio/skills` or `/agents` with the collected fields; skill/agent created with `enabled: false` per T001/T004 default) and **Edit first** (returns to the form for manual review). Archie NEVER saves without this explicit confirmation. On successful save, Archie posts a final message: `"Done! Your {skill|agent} '{name}' is created and disabled. Test the handler, then enable it."`
